## interface와 clocking block

`ram_if.sv`의 목적은 DUT 핀을 UVM component가 직접 만지지 않도록 중간 접점을 만드는 것이다.

| 구분 | 현재 코드 기준 역할 |
| :--- | :--- |
| `drv_cb` | driver 관점의 clocking block. `write`, `addr`, `wdata`는 output, `rdata`는 input이다. |
| `mon_cb` | monitor 관점의 clocking block. `write`, `addr`, `wdata`, `rdata`를 모두 input으로 관찰한다. |
| `default input #1step output #0` | clock edge 주변 race를 줄이기 위해 sampling/drive 시점을 clocking block에서 정의한다. |
| `modport DRV`, `modport MON` | driver와 monitor의 접근 방향을 분리한다. |

현재 interface에는 `read`나 `valid` 신호가 없다. 따라서 monitor는 `write == 1`이면 write, `write == 0`이면 read 후보로 해석한다. 0611 오전에 교수님 최신 코드로 다시 가져온 버전에서는 `interface ram_if (input logic clk);` 내부의 중복 `logic clk;` 선언이 제거되어 기존 `Identifier previously declared` warning은 사라졌다.

## sequence 흐름

`ram_base_seq`는 `do_write(addr, data)`와 `do_read(addr)` task를 제공한다.

| sequence | 목적 |
| :--- | :--- |
| `ram_wr_rd_seq` | random address에 먼저 write하고, queue에 저장한 같은 address들을 다시 read한다. write 후 readback 기본 시나리오다. |
| `ram_random_seq` | `write dist {1 := 6, 0 := 4}`로 write 비율을 더 높게 두고, `wdata inside {[8'h00 : 8'h10]}`으로 write data 범위를 제한한 random transaction을 반복한다. 현재 `ram_basic_test`에서 `num = 5000`으로 실행된다. |
| `ram_corner_seq` | `0x00`, `0xFF`, overwrite 등 corner address/data를 확인하기 위한 시퀀스다. 현재 test에서는 주석 처리되어 있다. |

수업 마지막 기준 `ram_basic_test`는 `ram_wr_rd_seq`와 `ram_corner_seq` 실행 코드를 주석 처리하고, `ram_random_seq`만 실행한다.

## 0611 오전 보강: sequence와 test scenario 관점

0611 오전에는 0610 RAM 코드에서 설명이 부족했던 부분을 다시 보정했다. 전날까지는 driver/monitor 연결과 scoreboard 동작에 관심이 많이 갔지만, 실제 검증에서는 sequence가 어떤 test scenario를 만들고 DUT spec의 어떤 동작을 확인하는지가 중요하다고 정리했다.

RAM DUT를 기준으로 보면 spec에는 최소한 `write`, `addr`, `wdata`, `rdata` 동작이 있다. 검증 시나리오는 이 신호 조합으로 어떤 동작을 확인할 것인지 정하는 과정이다. 단순히 random item을 많이 보내는 것이 아니라, "어떤 주소에 어떤 데이터를 쓰고, 어떤 시점에 다시 읽어서 무엇을 확인할 것인가"가 먼저 정리되어야 한다.

`ram_base_seq`의 `do_write`와 `do_read`는 반복되는 절차를 task로 묶은 것이다. write는 `write=1`, `addr`, `wdata`를 item에 넣고, read는 `write=0`, `addr`를 item에 넣는다. 이 공통 task를 두면 random sequence, corner sequence, direct sequence가 같은 write/read 절차를 재사용할 수 있다.

`start_item(item)`과 `finish_item(item)`의 의미도 다시 정리했다.

| 호출 | 의미 |
| :--- | :--- |
| `start_item(item)` | sequencer에게 item 전송을 준비했다고 알리고, sequencer가 grant할 때까지 대기하는 구간이다. 이 호출 자체가 driver로 item을 실제 전송하는 시점은 아니다. |
| item field 설정/randomize | scenario에 맞는 값을 item에 채우는 구간이다. 예를 들어 `write dist`, `wdata inside`, address 범위 등을 여기서 결정한다. |
| `finish_item(item)` | item 설정이 끝났음을 알리고 sequencer/driver 흐름으로 transaction이 넘어가게 하는 구간이다. |

면접이나 발표에서 UVM 검증을 설명할 때는 "driver와 monitor를 만들었다"에서 끝나면 약하다. 어떤 DUT spec을 기준으로 어떤 test scenario를 만들었고, 그 scenario를 sequence item에 반영해 어떤 조건을 검증했는지를 말해야 한다. 예를 들어 RAM에서는 기본 write/readback, 여러 주소 접근, 경계 주소, 특수 데이터, 같은 주소 반복 write/read, random test와 direct test의 목적을 구분해서 설명할 수 있어야 한다.

기능 테스트와 비기능 테스트도 구분했다. 지금 RAM UVM 예제는 기본 기능 테스트에 가깝다. 즉 주소에 쓰고 읽었을 때 기대값이 맞는지를 확인한다. 반면 비기능 테스트는 성능, 장시간 반복, 환경 조건 같은 관점까지 포함한다. 예를 들어 같은 주소에 장시간 반복 write/read를 수행하거나, 속도 조건에서 동작이 유지되는지 확인하는 식이다.

검증 엔지니어의 역할은 pass 로그를 만드는 것이 아니라 bug를 찾을 수 있는 경우의 수를 설계하는 것이다. 그래서 corner case와 direct test를 별도로 만들어야 한다. 대표적인 corner case는 최소 주소 `0x00`, 최대 주소 `0xFF`, 특수 데이터 `0x00`, `0xFF`, overwrite, 같은 주소 반복 접근 등이다. random test만으로는 특정 bug를 안정적으로 찌르기 어려울 수 있으므로, 의도한 값을 직접 넣는 direct test도 필요하다.

coverage는 "내가 생각한 경우의 수 중 몇 퍼센트를 실제로 테스트했는지"를 보는 지표다. 따라서 coverage가 낮으면 random 횟수만 늘릴지, constraint를 바꿀지, direct/corner sequence를 추가할지 판단해야 한다. 최신 코드에서 `wdata inside {[8'h00 : 8'h10]}`으로 값을 제한한 결과 `data_max = 8'hFF` bin이 hit되지 않은 것이 그 예다.

## monitor 동작

monitor의 핵심 구조는 다음과 같다.

```systemverilog
if (pending_rd != null) begin
    pending_rd.rdata = r_if.mon_cb.rdata;
    ap.write(pending_rd);
    pending_rd = null;
end

tr.write = r_if.mon_cb.write;
tr.addr  = r_if.mon_cb.addr;
tr.wdata = r_if.mon_cb.wdata;

if (tr.write) begin
    ap.write(tr);
end else begin
    pending_rd = tr;
end
```

write transaction은 즉시 publish한다. read transaction은 synchronous RAM 특성 때문에 현재 cycle에서 address/write를 저장해 두고, 다음 clocking event에서 `rdata`를 붙여 publish한다.

주의할 점은 `write == 0`을 무조건 read 후보로 보는 구조라는 것이다. interface에 `read` 또는 `valid`가 없어서 idle cycle과 read cycle을 monitor만으로는 명확히 구분하지 못한다.

## scoreboard 정책

최신 교수님 코드는 scoreboard에 `written[256]`과 `skipped_count`를 추가했다.

```systemverilog
if (tr.write) begin
    write_count++;
    mem_model[tr.addr] = tr.wdata;
    written[tr.addr]   = 1;
end else begin
    read_count++;
    if (!written[tr.addr]) begin
        skipped_count++;
        return;
    end
    if (tr.rdata === mem_model[tr.addr]) begin
        pass_count++;
    end else begin
        fail_count++;
    end
end
```

이 구조는 write된 적 없는 주소의 read 후보를 fail로 보지 않고 skip 처리한다. 즉 monitor가 idle/read-before-write 후보를 read처럼 보냈더라도, scoreboard에서 `written[addr] == 0`이면 비교하지 않는다.

현재 코드에서는 `read_count++`가 skip 검사보다 앞에 있다. 따라서 `read count`는 실제 비교 read와 skipped read 후보를 합친 값으로 해석해야 한다.

## functional coverage

`ram_coverage.sv`는 `uvm_subscriber #(ram_seq_item)` 형태이며, `write(ram_seq_item t)`에서 transaction을 받아 covergroup을 sampling한다.

| coverpoint | 의미 |
| :--- | :--- |
| `cp_op` | `tr.write` 기준 read/write 동작 종류를 확인한다. |
| `cp_addr` | address를 `addr_min`, low, mid, high, `addr_max` 구간으로 나눠 확인한다. |
| `cp_wdata` | write 동작일 때만 `wdata`가 `0x00`, `0xFF`, 기타 범위를 hit하는지 확인한다. |
| `cx_op_addr` | read/write 동작과 address 구간의 조합을 cross로 확인한다. |

`report_phase`에서는 전체 coverage와 각 coverpoint/cross coverage를 출력하고, 전체 coverage가 100% 미만이면 warning을 낸다.

## 수업 중 확인한 이슈

| 항목 | 정리 |
| :--- | :--- |
| 교수님 원본 보존 | 현재 `260610_ram`은 교수님 최신 코드 그대로 둔다. 수업 중 실험성 수정은 반영하지 않았다. |
| `bit`/`logic` 타입 | DUT와 interface는 `logic`인데 `ram_seq_item`은 `bit`이다. monitor에서 `logic` 값을 item의 `bit` 필드로 복사하면 X/Z가 0으로 눌릴 수 있다. 수업 후 리팩토링에서는 item도 `logic`으로 맞추는 것을 검토한다. |
| read/idle 구분 | interface에 `read`나 `valid`가 없어 monitor가 `write == 0`을 read 후보로 본다. 구조적으로는 `read` 또는 `valid/en` 신호를 추가하는 방식이 더 명확하다. |
| scoreboard skip | 교수님 코드는 `written[addr]`로 write-before-read 여부를 확인하고, write된 적 없는 주소는 `skipped_count`로 분리한다. |
| `read_count++` 위치 | 실제 비교된 read만 세고 싶다면 `read_count++`를 `if (!written[tr.addr]) return;` 이후로 옮긴다. |
| interface clock 선언 | 0611 오전 최신 코드에서 `interface ram_if (input logic clk);` 내부의 중복 `logic clk;` 선언이 제거되었다. 이 수정으로 기존 `Identifier previously declared` warning은 해결되었다. |
| `function write` return warning | `ram_scoreboard.sv`의 `function write(ram_seq_item tr);`에서 `return;`을 사용해 `Return expression omitted` warning이 나온다. 수업 후에는 `function void write(ram_seq_item tr);` 형태로 정리할 수 있다. |
| coverage 해석 | coverage는 coverpoint/bin hit 여부를 뜻한다. 최신 실행에서는 `wdata` constraint 때문에 데이터 coverage가 `66.67%`로 남았고 전체 coverage가 `91.67%`가 되었다. read/idle protocol이 구조적으로 정리되었는지는 scoreboard 정책과 함께 별도로 봐야 한다. |
