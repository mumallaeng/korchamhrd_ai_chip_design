## RTL과 TB를 나누는 기준

| 구문 | RTL | TB | 메모 |
| --- | --- | --- | --- |
| `always @(*)` | 가능 | 가능 | 조합논리 |
| `for` | 가능 | 가능 | RTL에서는 반복 범위가 고정돼야 읽기 쉽다 |
| `@(posedge clk)` | 가능 | 가능 | 순차 로직이나 TB 동기화 |
| `@(negedge clk)` | 보통 TB | 가능 | 테스트 순서 제어에 자주 쓴다 |
| `#10` | 보통 TB | 가능 | 지연 시간 제어 |
| `force/release` | 불가 | 가능 | 검증 편의를 위한 강제 구동 |

## 코드로 보면 이렇게 다르다

```verilog
always @(*) begin
    case (sel)
        2'b00: mux_out = a;
        2'b01: mux_out = b;
        default: mux_out = 4'b0000;
    endcase
end
```

```verilog
@(negedge clk);
#10;
wait (hour == 5'd23);
```

## blocking / nonblocking

| 구분 | 예시 위치 | 핵심 |
| --- | --- | --- |
| blocking `=` | `tb_practice_1.v:11-16` | 같은 절차 안에서 바로 값이 바뀐 것처럼 읽힌다 |
| nonblocking `<=` | `tb_practice_1.v:18-24` | 현재 시점 계산 후 다음 업데이트 시점에 반영된다 |
| 권장 | 순차 `always @(posedge clk)`에서는 `<=`를 기본으로 본다 |

## 주의점

| 실수 | 정리 |
| --- | --- |
| `#delay`를 RTL 타이밍 설계처럼 생각한다 | 합성되는 hardware delay와는 다르다 |
| testbench 문법을 RTL에도 그대로 써도 된다고 본다 | `force`, `wait`, 파일 dump는 검증 전용이다 |
| blocking과 nonblocking을 섞어 써도 된다고 본다 | 한 블록 안에서는 의도를 분명히 나누는 편이 안전하다 |
