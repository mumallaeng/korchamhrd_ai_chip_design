## 이 장의 핵심

- 구조적 모델링은 `이미 만든 모듈을 상위 모듈에서 조립해 더 큰 회로를 만드는 방식`이다.
- 핵심 단위는 `module`, `instance`, `port`, `wire`, `parameter`, `generate`다.
- 상위 모듈은 보통 "연산"보다 `연결`과 `구성`의 책임을 가진다.
- 구조적 모델링은 `재사용`, `디버깅`, `계층적 분할`, `팀 작업`에 강하다.
- 이번 수업의 `4bit adder -> 8bit adder`, `digit_splitter + mux + decoder + counter -> fnd_controller`가 대표 예시다.

## 한 줄 요약

행위수준 모델링이 "이 회로가 무엇을 하느냐"를 코드로 표현하는 쪽이라면, 구조적 모델링은 "이 회로가 어떤 하위 블록들로 구성되느냐"를 `연결도` 관점으로 표현하는 쪽이다.

## 현재 수업과 연결

이번 강의 흐름에서 6장은 아래 내용을 이해하기 위한 중심 장이다.

- `full_adder_4bit`를 재사용해 `full_adder_8bit`를 만드는 것
- `digit_splitter`, `mux_4x1`, `decoder_2x4`, `counter_4`를 묶어 `fnd_controller`를 만드는 것
- 큰 설계를 한 번에 쓰지 않고 `작은 검증 가능한 모듈`로 쪼개는 것
- 상위 모듈에서 `clk`, `rst`, `data`, `sel`, `fnd_com`, `fnd_data`를 배선하는 것

즉 이 장은 단순 문법 장이 아니라, 이후 FPGA 실습 전체를 지탱하는 `설계 조직화 방법` 장이다.

## 1. 구조적 모델링이 필요한 이유

### 1.1 한 파일에 모든 로직을 몰아 쓰면 생기는 문제

- 코드가 길어지면 읽기 어렵다.
- 일부 기능만 다시 쓰기 어렵다.
- 버그가 생겼을 때 어느 부분이 문제인지 분리하기 어렵다.
- 팀 작업에서 경계가 불분명해진다.

### 1.2 구조적 모델링의 장점

- 작은 모듈 단위로 검증할 수 있다.
- 이미 만든 블록을 그대로 재사용할 수 있다.
- 상위 모듈에서 전체 연결 구조를 한눈에 볼 수 있다.
- 설계 문서, 블록 다이어그램, 테스트벤치 구성과 잘 맞는다.

### 1.3 수업식으로 이해하기

- `1bit full adder` 하나를 먼저 만든다.
- 그것을 4개 이어서 `4bit adder`를 만든다.
- `4bit adder` 두 개를 다시 이어서 `8bit adder`를 만든다.

이 과정이 바로 구조적 모델링이다.  
핵심은 "큰 회로를 새로 발명한다"가 아니라, `검증된 블록을 위로 조립한다`는 것이다.

## 2. 기본 용어 정리

### 2.1 module

회로 블록의 설계도다. 입력과 출력, 내부 동작을 정의한다.

### 2.2 instance

모듈 설계도를 실제로 회로 안에 배치해 쓰는 실체다.  
같은 모듈도 인스턴스를 여러 개 만들 수 있다.

### 2.3 port

모듈 바깥과 통신하는 입출력 연결점이다.

### 2.4 wire

모듈과 모듈 사이를 잇는 배선이다. 구조적 모델링에서는 `wire`가 특히 자주 등장한다.

### 2.5 hierarchy

상위 모듈과 하위 모듈의 계층 관계다.

예를 들면 다음과 같다.

```text
top
├─ adder_8bit
│  ├─ full_adder_4bit u_low
│  └─ full_adder_4bit u_high
└─ fnd_controller
   ├─ digit_splitter
   ├─ mux_4x1
   ├─ bcd_decoder
   └─ decoder_2x4
```

## 3. 모듈 정의 문법

### 문법만 먼저 보기

```verilog
module module_name (
    input  ...,
    output ...
);
    // internal logic
endmodule
```

### 가장 기본적인 형태

가장 기본적인 모듈 정의 형태는 다음과 같다.

```verilog
module module_name (
    input        a,
    input        b,
    output       y
);

    assign y = a & b;

endmodule
```

### 해석

- `module`로 시작해서 `endmodule`로 끝난다.
- 괄호 안에 포트를 선언한다.
- 내부에는 `assign`, `always`, 하위 모듈 인스턴스 등을 둘 수 있다.

## 4. 인스턴스화와 포트 연결

### 문법만 먼저 보기

위치 기반 연결:

```verilog
child_module instance_name (sig0, sig1, sig2, sig3);
```

이름 기반 연결:

```verilog
child_module instance_name (
    .port0(sig0),
    .port1(sig1),
    .port2(sig2)
);
```

### 4.1 위치 기반 연결

포트 순서를 정확히 기억하고 있을 때는 위치 기반 연결이 가능하다.

```verilog
half_adder u0 (a, b, sum, carry);
```

하지만 이 방식은 순서를 헷갈리기 쉽다. 포트가 많아질수록 위험하다.

### 4.2 이름 기반 연결

실무와 학습 모두에서 더 안전한 방식이다.

```verilog
half_adder u0 (
    .a(a),
    .b(b),
    .sum(sum),
    .carry(carry)
);
```

### 이름 기반 연결을 선호하는 이유

- 포트 순서를 외우지 않아도 된다.
- 읽는 사람이 연결 의미를 바로 알 수 있다.
- 포트 순서가 바뀌어도 코드가 덜 깨진다.

## 5. 예제 1: 1비트 전가산기

구조적 모델링은 작은 블록부터 출발하는 것이 좋다.  
먼저 1비트 전가산기를 만든다.

### 문법만 먼저 보기

```verilog
module child_name (
    input  ...,
    output ...
);
    assign ...;
endmodule
```

### 예제

```verilog
module full_adder_1bit (
    input  a,
    input  b,
    input  cin,
    output sum,
    output cout
);
    assign sum  = a ^ b ^ cin;
    assign cout = (a & b) | (b & cin) | (a & cin);
endmodule
```

### 포인트

- 이 모듈은 내부 배선 없이도 `assign`만으로 정의 가능하다.
- 이후 상위 모듈에서 여러 번 재사용할 수 있다.

## 6. 예제 2: 4비트 가산기를 구조적으로 만들기

이제 1비트 전가산기 4개를 이어서 4비트 가산기를 만든다.

### 문법만 먼저 보기

```verilog
module parent_name (...);
    wire internal_sig0, internal_sig1;

    child_module u0 (...);
    child_module u1 (...);
    child_module u2 (...);
endmodule
```

### 예제

```verilog
module full_adder_4bit (
    input  [3:0] a,
    input  [3:0] b,
    input        cin,
    output [3:0] sum,
    output       cout
);
    wire c1, c2, c3;

    full_adder_1bit u0 (
        .a   (a[0]),
        .b   (b[0]),
        .cin (cin),
        .sum (sum[0]),
        .cout(c1)
    );

    full_adder_1bit u1 (
        .a   (a[1]),
        .b   (b[1]),
        .cin (c1),
        .sum (sum[1]),
        .cout(c2)
    );

    full_adder_1bit u2 (
        .a   (a[2]),
        .b   (b[2]),
        .cin (c2),
        .sum (sum[2]),
        .cout(c3)
    );

    full_adder_1bit u3 (
        .a   (a[3]),
        .b   (b[3]),
        .cin (c3),
        .sum (sum[3]),
        .cout(cout)
    );
endmodule
```

### 여기서 꼭 이해할 점

- `u0`, `u1`, `u2`, `u3`는 같은 모듈의 서로 다른 인스턴스다.
- `c1`, `c2`, `c3`는 각 인스턴스를 잇는 중간 carry 배선이다.
- 상위 모듈은 계산을 새로 쓰지 않고 `연결만` 한다.

즉 구조적 모델링의 본질은 `회로의 조립`이다.

## 7. 예제 3: 8비트 가산기를 더 큰 계층으로 만들기

이제 4비트 가산기 두 개를 이어서 8비트 가산기를 만든다.

### 문법만 먼저 보기

```verilog
module top_name (...);
    wire middle_signal;

    lower_block u_low (..., .cout(middle_signal));
    upper_block u_high(..., .cin (middle_signal));
endmodule
```

### 예제

```verilog
module full_adder_8bit (
    input  [7:0] a,
    input  [7:0] b,
    input        cin,
    output [7:0] sum,
    output       cout
);
    wire c_mid;

    full_adder_4bit u_low (
        .a   (a[3:0]),
        .b   (b[3:0]),
        .cin (cin),
        .sum (sum[3:0]),
        .cout(c_mid)
    );

    full_adder_4bit u_high (
        .a   (a[7:4]),
        .b   (b[7:4]),
        .cin (c_mid),
        .sum (sum[7:4]),
        .cout(cout)
    );
endmodule
```

### 수업 연결 포인트

이 예제가 Day 3 강의에서 본 `4bit -> 8bit 확장`의 전형적인 구조다.

- 한 번 검증한 `full_adder_4bit`를 그대로 쓴다.
- 상위 모듈은 carry만 중간에 이어 준다.
- 이런 식으로 설계를 위로 쌓아 올리는 것이 구조적 모델링의 핵심 감각이다.

## 8. 상위 모듈은 "연결 책임"을 가진다

상위 모듈(top module)은 보통 다음 일을 한다.

- 외부 입력 포트 선언
- 내부 `wire` 선언
- 하위 모듈 인스턴스화
- 하위 모듈들 사이 연결
- 보드 핀과 직접 연결되는 최종 출력 정리

### 문법만 먼저 보기

```verilog
module top (...);
    wire sig_a;
    wire sig_b;

    block_a u_a (..., .out(sig_a));
    block_b u_b (.in(sig_a), .out(sig_b));
    block_c u_c (.in(sig_b), ...);
endmodule
```

예를 들어 FND 제어기에서는 이런 블록 연결이 상위 모듈의 역할이다.

```text
input_data
   │
   ▼
digit_splitter ──> digit0, digit1, digit2, digit3
                        │
                        ▼
                    mux_4x1 ──> selected_digit ──> bcd_decoder ──> fnd_data
                        ▲
                        │
                    counter_4 ──> sel ──> decoder_2x4 ──> fnd_com
```

이때 `digit_splitter`, `mux_4x1`, `bcd_decoder`, `decoder_2x4`, `counter_4` 각각은 독립 검증 가능한 하위 모듈이 된다.

## 9. 포트 연결 시 자주 하는 실수

### 9.1 입력과 출력을 반대로 연결

가장 흔한 실수다. 이름 기반 연결로 줄일 수 있다.

### 9.2 비트 폭이 안 맞는 연결

예:

```verilog
.a(a[7:0])   // 8비트
.b(b[3:0])   // 4비트
```

폭이 다르면 합성/시뮬레이션에서 경고가 나거나 의도와 다르게 잘린다.

### 9.3 중간 배선을 선언하지 않음

하위 모듈 출력을 다른 하위 모듈 입력으로 넘기려면 중간 `wire`가 필요하다.

### 9.4 위치 기반 연결 순서 실수

포트가 많을수록 위험하다. 이름 기반 연결을 기본으로 삼는 편이 낫다.

## 10. 계층적 설계

계층적 설계는 복잡한 시스템을 단계별 블록으로 나누는 방식이다.

### 10.1 왜 계층을 나누는가

- 한 번에 이해할 범위를 줄이기 위해
- 기능별 책임을 분리하기 위해
- 재사용과 교체를 쉽게 하기 위해

### 10.2 Verilog에서 계층이 보이는 위치

- 소스 코드의 인스턴스 구조
- 시뮬레이터 파형 창의 트리 구조
- 합성 후 RTL schematic

즉 계층은 코드에서도, 툴에서도, 디버깅 과정에서도 중요하다.

## 11. 계층적 이름 참조

하위 모듈 내부 신호를 계층 경로로 참조하는 방식이다.

### 문법만 먼저 보기

```verilog
tb.dut.internal_signal
tb.dut.u_child.internal_signal
tb.dut.GEN_BLOCK[0].u_cell.internal_signal
```

예를 들면 테스트벤치에서 이런 식으로 볼 수 있다.

```verilog
$display("carry mid = %b", dut.u_low.cout);
```

또는 내부 `wire`를 직접 볼 수도 있다.

```verilog
$display("internal carry = %b", dut.c_mid);
```

### 주의점

- 계층적 이름 참조는 디버깅과 테스트벤치에는 유용하다.
- 하지만 설계 모듈끼리 강하게 의존하게 만드는 방식으로 남용하면 좋지 않다.
- 하위 모듈 내부 신호를 외부에서 자주 참조해야 한다면, 애초에 포트 설계를 다시 보는 편이 낫다.

즉 `계층적 이름`은 기본 설계 인터페이스가 아니라 `관찰 도구`에 가깝다.

## 12. `parameter`를 이용한 재사용

비슷한 구조를 비트 폭만 바꿔 여러 번 쓰고 싶을 때 `parameter`를 사용한다.

### 문법만 먼저 보기

```verilog
module module_name #(
    parameter WIDTH = 8
) (
    input  [WIDTH-1:0] in,
    output [WIDTH-1:0] out
);
endmodule

module_name #(
    .WIDTH(16)
) u0 (
    .in (...),
    .out(...)
);
```

### 12.1 예제: parameter를 가진 모듈 정의

```verilog
module register_nbit #(
    parameter WIDTH = 8
) (
    input              clk,
    input              rst,
    input  [WIDTH-1:0] d,
    output reg [WIDTH-1:0] q
);
    always @(posedge clk or posedge rst) begin
        if (rst)
            q <= {WIDTH{1'b0}};
        else
            q <= d;
    end
endmodule
```

### 12.2 예제: 인스턴스에서 파라미터 변경

```verilog
register_nbit #(
    .WIDTH(4)
) u_reg4 (
    .clk(clk),
    .rst(rst),
    .d  (d4),
    .q  (q4)
);

register_nbit #(
    .WIDTH(16)
) u_reg16 (
    .clk(clk),
    .rst(rst),
    .d  (d16),
    .q  (q16)
);
```

### 왜 중요한가

- 같은 모듈을 4비트, 8비트, 16비트용으로 반복 작성하지 않아도 된다.
- 수정 포인트가 줄어든다.
- 구조적 모델링과 궁합이 좋다.

## 13. `generate`를 이용한 반복 구조 생성

`generate`는 "같은 패턴의 인스턴스를 여러 개 만들기" 위한 문법이다.  
특히 비트 수가 커질수록 손으로 인스턴스를 반복 작성하는 대신 많이 쓴다.

### 문법만 먼저 보기

```verilog
genvar i;

generate
    for (i = 0; i < N; i = i + 1) begin : GEN_NAME
        child_module u_child (
            .in (...),
            .out(...)
        );
    end
endgenerate
```

### 13.1 핵심 개념

- `generate`는 시뮬레이션 중에 반복 실행되는 문법이 아니다.
- `elaboration` 시점에 하드웨어 구조를 만들어 내는 문법이다.
- 즉 `for-generate`는 "코드를 반복 실행"한다기보다 `회로 인스턴스를 여러 개 펼친다`.

### 13.2 예제: ripple carry adder

```verilog
module ripple_adder #(
    parameter WIDTH = 8
) (
    input  [WIDTH-1:0] a,
    input  [WIDTH-1:0] b,
    input              cin,
    output [WIDTH-1:0] sum,
    output             cout
);
    wire [WIDTH:0] c;
    genvar i;

    assign c[0] = cin;
    assign cout = c[WIDTH];

    generate
        for (i = 0; i < WIDTH; i = i + 1) begin : GEN_FA
            full_adder_1bit u_fa (
                .a   (a[i]),
                .b   (b[i]),
                .cin (c[i]),
                .sum (sum[i]),
                .cout(c[i+1])
            );
        end
    endgenerate
endmodule
```

### 여기서 볼 포인트

- `genvar i`는 generate 전용 반복 변수다.
- `GEN_FA[0]`, `GEN_FA[1]`처럼 이름이 붙은 인스턴스 그룹이 만들어진다.
- `WIDTH`만 바꾸면 다른 폭의 가산기를 같은 구조로 만들 수 있다.

## 14. 구조적 모델링과 행위수준 모델링의 차이

### 구조적 모델링에 더 가까운 코드

- 하위 모듈 인스턴스를 여러 개 선언
- `wire`로 연결
- 상위 모듈은 조립 중심

### 행위수준 모델링에 더 가까운 코드

- `always`, `if`, `case`, `for`로 동작 기술
- 내부 알고리즘 또는 상태 전이를 직접 표현

### 실제 설계에서는 둘을 같이 쓴다

실전에서는 보통 아래처럼 섞인다.

- 하위 모듈 내부는 행위수준으로 작성
- 상위 모듈은 구조적으로 조립

예를 들어:

- `mux_4x1` 내부는 `always @(*)` + `case`
- `counter_4` 내부는 `always @(posedge clk or posedge rst)`
- 이 둘을 묶는 `fnd_controller`는 구조적 모델링

즉 하나만 고집하는 것이 아니라, `어느 계층에서 어떤 표현이 맞는가`를 고르는 것이 중요하다.

## 15. 테스트벤치에서 계층 구조를 보는 예

아래처럼 DUT 내부 중간 carry를 관찰할 수 있다.

```verilog
module tb_full_adder_8bit;
    reg  [7:0] a;
    reg  [7:0] b;
    reg        cin;
    wire [7:0] sum;
    wire       cout;

    full_adder_8bit dut (
        .a   (a),
        .b   (b),
        .cin (cin),
        .sum (sum),
        .cout(cout)
    );

    initial begin
        a   = 8'h0F;
        b   = 8'h01;
        cin = 1'b0;
        #10;

        $display("sum     = %h", sum);
        $display("cout    = %b", cout);
        $display("c_mid   = %b", dut.c_mid);
    end
endmodule
```

### 해석

- `dut.c_mid`는 상위 DUT 내부 `wire`다.
- 파형 창에서도 이런 내부 신호를 찾아볼 수 있다.
- 구조적 모델링을 하면 내부 연결을 추적하기가 쉬워진다.

## 16. 실전에서 바로 쓰는 구조적 모델링 체크리스트

### 모듈을 나눌 때

- 기능 경계가 분명한가
- 단독 테스트가 가능한가
- 다른 프로젝트에서도 재사용 가능한가

### 인스턴스화할 때

- 포트 폭이 맞는가
- 이름 기반 연결을 썼는가
- 중간 `wire` 선언이 빠지지 않았는가

### 상위 모듈을 쓸 때

- 상위 모듈이 계산을 너무 많이 떠안고 있지 않은가
- 블록도와 코드 구조가 대응되는가
- `clk`, `rst`, 제어신호의 흐름이 눈에 보이는가

### 테스트할 때

- 하위 모듈 단독 테스트를 먼저 했는가
- 상위 통합 테스트를 별도로 했는가
- 내부 배선이나 중간 캐리 같은 신호를 파형으로 확인했는가

## 17. 자주 하는 실수

### 17.1 모듈과 인스턴스를 구분하지 못함

- `full_adder_1bit`는 설계도
- `u0`, `u1`, `u2`, `u3`는 실체

이 둘을 섞어 생각하면 계층 구조가 흐려진다.

### 17.2 상위 모듈에서 모든 로직을 직접 작성함

이러면 구조적 모델링의 장점이 사라진다.  
상위 모듈은 가급적 배선과 조립 중심으로 유지하는 편이 좋다.

### 17.3 `parameter` 없이 비슷한 코드를 여러 개 복붙함

처음에는 빠른 것 같아도 수정 시 유지보수 비용이 크게 늘어난다.

### 17.4 `generate`를 일반 `always` 반복과 혼동함

`generate`는 하드웨어 구조 생성이고, `always` 안의 `for`는 절차적 기술이다. 목적이 다르다.

## 18. 이 장을 공부한 뒤 스스로 설명할 수 있어야 하는 것

- 구조적 모델링이 왜 필요한가
- 모듈과 인스턴스의 차이는 무엇인가
- 이름 기반 포트 연결이 왜 안전한가
- `4bit adder -> 8bit adder`가 왜 구조적 모델링 예시인가
- `parameter`와 `generate`가 재사용성을 어떻게 높이는가
- 계층적 이름 참조는 언제 유용하고 왜 남용하면 안 되는가

## 19. 복습 질문

1. `full_adder_1bit` 4개를 이어 `full_adder_4bit`를 만들 때 필요한 중간 배선은 무엇인가
2. `full_adder_4bit` 두 개를 이어 `full_adder_8bit`를 만들 때 carry는 어떻게 넘기는가
3. 이름 기반 연결과 위치 기반 연결 중 왜 이름 기반 연결이 더 안전한가
4. `parameter WIDTH = 8`이 있으면 어떤 이점이 생기는가
5. `generate`와 `always`의 반복은 어떤 점이 다른가
6. `dut.c_mid` 같은 계층적 이름 참조는 언제 쓰는 것이 적절한가

## 20. 실습 연결 포인트

이 장을 이해하면 아래 실습들이 훨씬 자연스럽게 보인다.

- `full_adder_4bit` 재사용 기반 `full_adder_8bit`
- `digit_splitter + mux_4x1 + decoder_2x4 + counter_4` 기반 `fnd_controller`
- `adder + fnd_controller + clk_divider` 기반 상위 시스템

즉 6장은 이후 조합/순차회로 실습의 준비 장이 아니라, `실습 구조를 읽는 눈`을 만드는 장이다.
