## 이 장의 핵심

- 조합회로는 현재 입력만으로 출력이 결정된다.
- 조합회로의 대표 형태로는 기본 논리 게이트, mux, encoder, decoder, random logic, 가산기/감산기, 비교기, ALU, LUT, 3-state bus 등이 있다.
- `assign`과 `always @(*)`는 조합회로를 표현하는 두 핵심 방식이다.
- 게이트 프리미티브 외에도 `assign`, `always @(*)`, `if`, `case`, 연산자식, 함수, 모듈 인스턴스화 등이 조합회로 설계에 널리 쓰인다.
- 실제 회로를 만들려면 논리 합성이 지원되는 구문만 써야 하며, testbench용 구문을 RTL에 섞으면 안 된다.
- `if`와 `case`의 해석 차이, 기본값 처리, full case 습관이 중요하다.
- `always` 감지신호 목록, `else/default`, 기본값 초기화가 빠지면 합성 전후 시뮬레이션 차이나 래치 추론이 생길 수 있다.
- 조합회로는 동작만 맞추는 것이 아니라, 논리 최적화가 잘 되고 가독성이 좋은 형태로 모델링해야 한다.
- 래치를 만들지 않는 코딩 스타일이 조합회로 설계의 핵심 품질 기준이다.

## 1. 조합회로란 무엇인가

조합회로는 현재 입력만으로 결과가 정해지는 회로다.

- 기억소자가 없다.
- 이전 클럭 상태를 저장하지 않는다.
- 입력이 바뀌면 출력도 조합 규칙에 따라 즉시 결정된다.

### 형식만 먼저 보기

```text
output = f(current inputs)
```

### 1.1 조합회로의 대표 형태

수업에서 말한 "조합회로의 형태"는 결국 현재 입력을 어떤 방식으로 가공하느냐에 따라 나뉜다.

- 기본 논리 게이트: `and`, `or`, `not`, `xor`, `nand`, `nor`, `xnor`
- 멀티플렉서 (`mux`): 여러 입력 중 하나를 선택
- 인코더 (`encoder`): 여러 입력 상태를 코드로 압축
- 디코더 (`decoder`): 코드 입력을 개별 출력으로 확장
- random logic: 여러 입력 조건을 임의 논리식으로 조합한 일반 조합논리
- 가산기 / 감산기: 산술 연산 수행
- 비교기: `==`, `>`, `<` 등 비교 결과 생성
- ALU: 산술과 논리 기능을 한 블록으로 묶은 구조
- lookup table (`LUT`): 입력 조합에 대해 정해진 출력을 매핑
- 3-state bus: 버스에 연결할지 끊을지 제어하는 구조

### 해석

- `mux`, `encoder`, `decoder`는 구조가 비교적 정형화된 조합회로다.
- `3-state bus driver`는 공유 버스 구조를 설명할 때 자주 등장한다.
- `random logic`은 정형 블록으로 딱 떨어지지 않는 일반 논리식을 뜻한다.
- FPGA에서는 많은 조합논리가 내부적으로 `LUT` 기반으로 구현된다.
- 강의 실습에서 본 FND 선택 로직, digit 분리, decoder, 조건 분기들은 대부분 이 범주에 들어간다.

### 1.2 조합회로 설계에 쓰는 Verilog HDL 구문

슬라이드에서 게이트 프리미티브 외에 같이 봐야 하는 Verilog HDL 표현은 아래처럼 정리할 수 있다.

- 게이트 프리미티브
  예: `and`, `or`, `not`, `xor`
- 연속 할당문 (`assign`)
  단순 논리식, 비교기, 산술식 표현에 적합
- 절차적 블록 (`always @(*)`)
  분기와 경우 나열이 많은 조합회로에 적합
- 선택 구문 (`if`, `if-else`, `case`)
  우선순위 선택, 디코딩, mux 구현에 자주 사용
- 연산자 기반 식
  산술 연산자, 비트 연산자, 논리 연산자, 비교 연산자, 시프트 연산자, 조건 연산자 `?:`
- 연결 / 반복 구문
  예: `{a, b}`, `{4{en}}`
- 함수 (`function`)
  반복되는 조합 연산을 식처럼 재사용할 때 유용
- 모듈 인스턴스화
  이미 만든 조합 블록을 상위 모듈에서 연결할 때 사용

### 해석

- "조합회로는 `assign`만 쓴다"가 아니라, 표현 방식이 여러 개 있다는 점이 중요하다.
- 단순한 식은 `assign`, 구조적 분기는 `always @(*)`, 반복 계산은 `function`, 블록 조립은 인스턴스화로 가는 식이다.
- 결국 어떤 구문을 쓰든 합성 결과가 현재 입력만의 함수가 되면 조합회로다.

### 1.3 합성 가능한 구문과 비합성 구문

조합회로를 RTL로 쓸 때는 "문법이 된다"와 "합성이 된다"를 구분해야 한다. 실제 하드웨어 모델링은 논리 합성이 지원되는 구문만 사용해야 한다.

### 합성 가능한 쪽으로 보는 구문

- 게이트 프리미티브 중 기본 논리 게이트
  예: `and`, `or`, `not`, `xor`
- `assign`
- `always @(*)`
- `if`, `if-else`, `case`
- 산술 / 비트 / 논리 / 비교 / 시프트 / 조건 연산자
- 연결과 반복
  예: `{a, b}`, `{4{en}}`
- `function`
- 모듈 인스턴스화

### 논리 합성이 지원되지 않거나 RTL에 쓰지 않는 구문

- `initial`
- 스위치 프리미티브
  예: `cmos`, `nmos`, `pmos`, `rnmos`, `rpmos`, `tran`, `rtran`, `tranif0`, `tranif1`, `rtranif0`, `rtranif1`
- `forever`
- `wait`
- `fork ... join`
- `force` / `release`
- 지연 제어 `#`

### 해석

- `initial`, `forever`, `wait`, `fork ... join`, `force/release`, `#delay`는 대표적인 testbench성 구문이다.
- 스위치 프리미티브는 회로 수준 시뮬레이션 관점에서는 문법적으로 존재하지만, 일반적인 RTL 합성 대상으로 보지 않는다.
- 즉 조합회로 설계에서는 "합성 가능한 조합논리 표현"과 "시뮬레이션 전용 제어 구문"을 반드시 구분해야 한다.

### 1.4 조합논리회로 모델링 시 유의사항

수업에서 특히 중요하게 봐야 하는 부분은 "조합논리처럼 보이는 코드"가 아니라, 실제로도 조합논리로 합성되고 시뮬레이션에서도 동일하게 해석되는 코드인지다.

### 감지신호 목록은 빠짐없이 잡아야 한다

`always` 구문으로 조합회로를 쓰는 경우에는 감지신호 목록에 회로 입력이 빠짐없이 포함되어야 한다.

```verilog
always @(*) begin
    ...
end
```

### 해석

- 예전 방식처럼 sensitivity list를 수동으로 쓰면 입력 하나가 빠질 수 있다.
- 그러면 시뮬레이션에서는 어떤 입력 변화에 블록이 다시 실행되지 않아 출력이 늦게 갱신될 수 있다.
- 하지만 합성된 실제 하드웨어는 입력 논리식대로 반응하므로, 합성 전 시뮬레이션과 합성 후 동작 해석이 달라질 수 있다.
- 그래서 강의에서도 조합논리는 사실상 `always @(*)`로 쓰는 습관을 강조한다.

즉 "무조건 회로가 아예 안 돈다"기보다, 특정 입력 변화에서만 시뮬레이션 결과가 어긋나는 식의 문제가 생길 수 있어 더 위험하다.

### 모든 입력 조건에 대한 출력값을 써야 한다

`if` 조건문과 `case` 문에서도 가능한 입력 조건에 대한 출력 할당이 빠지지 않아야 한다.

```verilog
always @(*) begin
    y = 1'b0;
    if (en)
        y = a;
    else
        y = b;
end
```

또는

```verilog
always @(*) begin
    out = 4'b0000;
    case (sel)
        2'b00: out = in0;
        2'b01: out = in1;
        2'b10: out = in2;
        2'b11: out = in3;
        default: out = 4'b0000;
    endcase
end
```

### 해석

- `else`나 `default`가 없고, 블록 초기에 기본값도 주지 않으면 일부 조건에서 출력이 할당되지 않는다.
- 이 경우 합성기는 "이전 값을 기억해야 한다"로 해석할 수 있고, 그 결과 래치가 추론될 수 있다.
- 즉 의도는 조합회로였는데, 결과적으로 순차적인 기억 요소가 생기는 상황이 발생한다.

이 역시 항상 즉시 오동작하는 식으로 드러나는 건 아니다. 어떤 입력 패턴에서는 멀쩡해 보일 수 있고, 특정 조건에서만 이전 값이 남아 문제가 드러날 수 있다. 그래서 더 조심해야 한다.

### 기본값, `else`, `default`를 습관처럼 둔다

조합논리에서는 아래 세 가지가 사실상 안전장치다.

- 블록 시작 시 출력 기본값 지정
- `if` 체인에 대한 `else`
- `case` 문에 대한 `default`

이 셋을 잘 써두면 불완전 할당으로 인한 래치 추론을 크게 줄일 수 있다.

### 논리 최적화와 가독성을 같이 본다

조합회로 모델링은 문법만 맞는다고 끝이 아니다. 합성 결과가 최소 게이트 수, 짧은 지연 경로 쪽으로 잘 정리되도록 쓰는 감각도 중요하다.

- 논리 최적화가 쉬운 구조를 사용한다.
- 불필요하게 장황한 중복 분기보다 간결한 식을 우선한다.
- 읽기 쉬운 이름과 구조를 사용한다.
- 가독성이 좋아야 오류를 줄이고 디버깅도 쉬워진다.

### 해석

- 같은 기능이라도 표현 방식에 따라 합성기가 최적화하기 쉬운 코드와 어려운 코드가 있다.
- 너무 복잡한 중첩 `if`나 불필요한 임시 신호 남발은 읽기도 어렵고 디버깅도 힘들다.
- 결국 좋은 조합회로 코드는 "의도한 기능", "합성 안정성", "최적화 가능성", "가독성"을 함께 만족해야 한다.

## 2. `assign`으로 쓰는 조합회로

단순 논리식에는 `assign`이 가장 간결하다.

### 문법만 먼저 보기

```verilog
assign y = expr;
```

### 예제

```verilog
module comparator_eq (
    input  [3:0] a,
    input  [3:0] b,
    output       eq
);
    assign eq = (a == b);
endmodule
```

### 해석

- 비교기처럼 식이 단순할수록 `assign`이 읽기 쉽다.
- 조합회로라는 사실이 한눈에 드러난다.

### 2.1 기본 논리 게이트 모델링 비교

수업에서는 기본 논리 게이트도 한 가지 방식만 있는 것이 아니라, 여러 Verilog 표현으로 모델링할 수 있다는 점을 비교했다. 대표 예로 `3-input NOR` 게이트를 네 방식으로 쓸 수 있다.

### 비트 연산자 사용

```verilog
module nor3_bitwise (
    input  a,
    input  b,
    input  c,
    output y
);
    assign y = ~(a | b | c);
endmodule
```

### 축약 연산자 사용

```verilog
module nor3_reduction (
    input  [2:0] in,
    output       y
);
    assign y = ~(|in);
endmodule
```

### 게이트 프리미티브 사용

```verilog
module nor3_primitive (
    input  a,
    input  b,
    input  c,
    output y
);
    nor (y, a, b, c);
endmodule
```

### `if` 조건문 사용

```verilog
module nor3_if (
    input  a,
    input  b,
    input  c,
    output reg y
);
    always @(*) begin
        if ({a, b, c} == 3'b000)
            y = 1'b1;
        else
            y = 1'b0;
    end
endmodule
```

### 해석

- 비트 연산자 방식은 논리식이 직접 드러나서 가장 기본적이다.
- 축약 연산자 방식은 입력 수가 벡터로 묶여 있을 때 간결하다.
- 게이트 프리미티브는 회로 기호 감각에 가깝지만, 실무 RTL에서는 연산자식이 더 자주 쓰인다.
- `if` 조건문 방식도 가능하지만, 단순 게이트만 놓고 보면 식이 길어져 가독성이 떨어질 수 있다.

즉 같은 `NOR` 게이트라도 표현은 여러 가지가 가능하고, 상황에 따라 가장 읽기 좋은 방식을 선택하면 된다.

### 시뮬레이션 testbench 감각

이 예제의 testbench는 `for`문으로 입력을 `000`부터 `111`까지 순차적으로 바꾸면서 모든 입력 조합을 확인하는 방식으로 이해하면 된다.

```verilog
module tb_nor3;
    reg a, b, c;
    integer i;

    initial begin
        for (i = 0; i < 8; i = i + 1) begin
            {a, b, c} = i[2:0];
            #10;
        end
    end
endmodule
```

### 해석

- `for`문은 testbench에서 가능한 입력 조합을 빠르게 훑을 때 매우 편하다.
- `3-input NOR`처럼 입력이 3개이면 총 `2^3 = 8`개 조합만 보면 된다.
- 이런 방식은 회로 자체를 합성하는 RTL이 아니라, 시뮬레이션 검증용 testbench에서 주로 사용한다.

### 2.2 멀티비트 논리연산 모델링

수업에서는 단일 게이트를 넘어서, 멀티비트 논리연산을 어떻게 모델링하는지도 가볍게 비교했다. 대표 예로 `4비트 2입력 XOR` 게이트를 세 방식으로 쓸 수 있다.

### 게이트 프리미티브 배열 사용

```verilog
module xor4_primitive_array (
    input  [3:0] a,
    input  [3:0] b,
    output [3:0] y
);
    xor g_xor[3:0] (y, a, b);
endmodule
```

### 비트 연산자 사용

```verilog
module xor4_assign (
    input  [3:0] a,
    input  [3:0] b,
    output [3:0] y
);
    assign y = a ^ b;
endmodule
```

### `always` 블록 안의 `for` 반복문 사용

```verilog
module xor4_for_loop (
    input  [3:0] a,
    input  [3:0] b,
    output reg [3:0] y
);
    integer i;

    always @(*) begin
        for (i = 0; i < 4; i = i + 1)
            y[i] = a[i] ^ b[i];
    end
endmodule
```

### 해석

- 게이트 프리미티브 배열은 "1비트 XOR 4개를 병렬로 놓는다"는 구조가 직접 보인다.
- `assign y = a ^ b;`는 가장 간결하고, 멀티비트 비트연산의 의미가 바로 드러난다.
- `always` 안의 `for`문은 비트별 반복 처리를 코드로 펼쳐 쓰는 느낌이다.

같은 기능이라면 실전 RTL에서는 보통 비트 연산자 방식이 가장 짧고 읽기 쉽다. 다만 구조를 교육용으로 보여주거나, 비트별 조건이 나중에 더 붙는 경우에는 `for` 반복문 방식도 자주 등장한다.

### `generate for`와 `always` 안의 `for` 차이

강의에서 언급한 포인트는 둘 다 반복처럼 보이지만 쓰임이 다르다는 점이다.

- `generate for`
  모듈/게이트/배선 인스턴스를 정적으로 반복 생성할 때 사용
- `always` 안의 `for`
  절차적 블록 안에서 반복적으로 할당문을 쓰는 방식

### 해석

- `generate for`는 회로 구조를 만들어 내는 쪽에 가깝다.
- `always @(*)` 안의 `for`는 조합논리 동작을 절차적으로 기술하는 쪽에 가깝다.
- 고정된 반복 횟수라면 둘 다 합성 가능할 수 있지만, 문법적 역할과 읽는 관점은 다르다.

즉 "같은 반복"처럼 보여도, `generate`는 구조 생성, `always` 안의 `for`는 동작 기술로 이해하는 편이 맞다.

### 테스트벤치 감각

이 부분의 테스트벤치는 아주 깊게 들어가진 않았고, 입력 벡터 몇 개를 바꿔 보면서 결과를 확인하는 정도로 가볍게 넘어갔다고 보면 된다.

```verilog
module tb_xor4;
    reg  [3:0] a, b;
    wire [3:0] y;

    initial begin
        a = 4'b0000; b = 4'b0000; #10;
        a = 4'b1010; b = 4'b0101; #10;
        a = 4'b1111; b = 4'b0011; #10;
        a = 4'b1100; b = 4'b1010; #10;
    end
endmodule
```

### 해석

- 멀티비트 논리연산은 한 비트씩 손으로 보지 않고, 벡터 단위로 기대 결과를 확인하는 연습이 중요하다.
- `4'b1010 ^ 4'b0101 = 4'b1111`처럼 바로 읽히는 감각이 필요하다.
- 수업에서는 이 예제를 통해 "멀티비트 연산은 같은 논리를 병렬 비트에 적용한 것"이라는 점을 확인한 것으로 보면 된다.

### 2.3 부울 함수와 진리표 및 lookup table의 모델링

수업에서는 기본 게이트나 정형 블록뿐 아니라, 부울 함수로 표현된 논리식과 진리표(`truth table`) 기반 회로를 코드로 옮기는 감각도 중요하다. 이때 `lookup table (LUT)` 관점으로 이해하면 정리가 쉽다.

### LUT란 무엇인가

`lookup table`은 회로 동작에 사용되는 고정 데이터 값을 저장해 두고, 입력 조합에 따라 대응되는 출력을 꺼내는 조합회로라고 볼 수 있다.

- 입력 조합을 주소처럼 본다.
- 미리 정해 둔 출력값을 읽어 낸다.
- 기억소자처럼 상태를 저장하는 순차회로가 아니라, 입력에 대한 고정 매핑을 수행하는 조합회로로 이해한다.

강의에서 다룬 예 중에서는 아래가 모두 LUT 감각으로 읽힌다.

- `ROM`
- `BCD decoder`
- `Binary-to-Gray` 변환

이런 회로는 진리표를 기준으로 정리한 뒤 `case` 문으로 옮기면 가장 읽기 쉬운 경우가 많다.

### 부울 함수 모델링

예를 들어 아래 식은 전체 식 위에 부정 바가 있는 형태로 읽는다.

$$
y = \overline{\left((a \cdot b) + (c \cdot d)\right) \cdot e}
$$

### 해석

- `\cdot` 는 AND
- `+` 는 OR
- 윗줄 바는 전체 식에 대한 NOT

즉 먼저 `(a \cdot b)`와 `(c \cdot d)`를 만든 뒤 OR 하고, 다시 `e`와 AND 한 다음, 마지막에 전체 결과를 반전하는 구조다.

### 비트 연산자를 사용한 Verilog 모델링

```verilog
module bool_func_example (
    input  a,
    input  b,
    input  c,
    input  d,
    input  e,
    output y
);
    assign y = ~(((a & b) | (c & d)) & e);
endmodule
```

### 해석

- 부울 식을 Verilog로 옮길 때는 보통
  `AND -> &`, `OR -> |`, `NOT -> ~`
  로 대응시킨다.
- 괄호를 충분히 써서 원래 부울 식의 우선순위가 그대로 보이게 하는 것이 좋다.
- 이런 식은 `assign`으로 쓰는 것이 가장 자연스럽고, 조합회로라는 점도 바로 드러난다.

### 진리표와 LUT 모델링 감각

부울 식이 아니라 입력-출력 대응표가 먼저 주어지는 경우에는 진리표를 그대로 코드로 옮기는 방식이 더 자연스럽다.

```verilog
always @(*) begin
    case (in)
        ...
        default: out = ...
    endcase
end
```

### 해석

- `case` 문은 "입력 조합마다 어떤 고정 출력을 내는가"를 나열하기 좋다.
- 그래서 `ROM`, `BCD decoder`, `Binary-to-Gray`처럼 고정 매핑 성격이 강한 회로와 잘 맞는다.
- 즉 부울 식이 간단하면 `assign`, 진리표 기반 매핑이 중심이면 `case` 기반 LUT 모델링으로 읽으면 된다.

## 3. `always @(*)`로 쓰는 조합회로

경우의 수가 많거나 분기 구조가 중요한 조합회로는 `always @(*)`가 더 낫다.

### 문법만 먼저 보기

```verilog
always @(*) begin
    out = default_value;
    case (sel)
        ...
    endcase
end
```

### 예제

```verilog
module decoder_2x4 (
    input  [1:0] sel,
    output reg [3:0] out
);
    always @(*) begin
        case (sel)
            2'b00: out = 4'b1110;
            2'b01: out = 4'b1101;
            2'b10: out = 4'b1011;
            2'b11: out = 4'b0111;
            default: out = 4'b1111;
        endcase
    end
endmodule
```

### 해석

- 7-segment FND 공통단자 선택처럼 경우의 수가 분명할 때 `case`가 자연스럽다.
- 강의에서 계속 강조한 `default`가 들어가 있다.

## 4. `if`와 `case`

둘 다 조합회로에 쓸 수 있지만 읽는 감각이 다르다.

- `if`: 우선순위가 있는 선택처럼 읽힌다.
- `case`: 나열된 선택지 중 하나를 고르는 느낌이 강하다.

### 문법만 먼저 보기

```verilog
if (en) y = a;
else    y = b;

case (sel)
    ...
endcase
```

### 해석

- 우선순위 인코더처럼 순서가 중요하면 `if`
- mux/decoder처럼 경우 나열이 자연스러우면 `case`

### 4.1 우선순위 인코더

우선순위 인코더(`priority encoder`)는 여러 입력이 동시에 활성화될 수 있을 때, 그중 어느 입력을 먼저 인정할지를 정하는 조합회로다.

예를 들면 아래처럼 읽는다.

- `in[3]`가 켜져 있으면 가장 높은 우선순위
- 그다음 `in[2]`
- 그다음 `in[1]`
- 마지막 `in[0]`

즉 여러 조건이 동시에 참일 수 있을 때는 "먼저 검사한 조건"이 우선순위를 가진다.

### 4:2 우선순위 인코더 진리표

여기서는 `in[3]`가 가장 높은 우선순위를 갖는다고 가정한다. 여러 입력이 동시에 `1`이어도 가장 높은 비트 하나만 인정한다.

| in[3:0] | out[1:0] | 설명 |
| --- | --- | --- |
| `0000` | `00` | 활성 입력 없음 |
| `0001` | `00` | `in[0]` 선택 |
| `001x` | `01` | `in[1]` 우선 |
| `01xx` | `10` | `in[2]` 우선 |
| `1xxx` | `11` | `in[3]` 최우선 |

### `if-else`로 쓰는 우선순위 인코더

우선순위를 명확하게 드러내려면 보통 `if-else` 체인이 가장 안전하다.

```verilog
module priority_encoder_4to2 (
    input  [3:0] in,
    output reg [1:0] out
);
    always @(*) begin
        out = 2'b00;

        if      (in[3]) out = 2'b11;
        else if (in[2]) out = 2'b10;
        else if (in[1]) out = 2'b01;
        else if (in[0]) out = 2'b00;
    end
endmodule
```

### `casex`를 이용한 우선순위 인코더

강의에서는 `casex`로도 같은 동작을 적을 수 있다는 점을 같이 보여준다.

```verilog
module priority_encoder_4to2_casex (
    input  [3:0] in,
    output reg [1:0] out
);
    always @(*) begin
        casex (in)
            4'b1xxx: out = 2'b11;
            4'b01xx: out = 2'b10;
            4'b001x: out = 2'b01;
            4'b0001: out = 2'b00;
            default: out = 2'b00;
        endcase
    end
endmodule
```

### `case` 문과 우선순위

`case` 문도 코드상으로는 나열된 항목을 위에서 아래 순서로 비교하고, 먼저 참이 되는 항목을 실행하는 흐름으로 이해할 수 있다. 특히 패턴이 서로 겹칠 수 있는 `casez`, `casex`에서는 이 순서성이 더 중요해진다.

하지만 실무에서는 아래를 구분해서 보는 것이 중요하다.

- 조건들이 서로 상호배타적이면
  우선순위가 사실상 필요 없고, 합성기는 병렬 선택 또는 mux 성격의 구조로 최적화하기 쉽다.
- 조건들이 서로 겹칠 수 있으면
  우선순위가 필요한 구조로 읽힐 수 있다.

### 해석

- 상호배타적인 선택은 `mux`처럼 이해하면 된다.
- 겹치는 조건이 있으면 "첫 번째 항목이 우선"이라는 의미가 실제 회로 구조에 영향을 줄 수 있다.
- 다만 이런 해석은 도구와 코딩 스타일에 따라 달라질 수 있으므로, 우선순위가 정말 중요하면 `if-else`로 명시적으로 쓰는 편이 가장 안전하다.

또한 `casex`는 `x`와 `z`까지 don't care처럼 처리하므로, 시뮬레이션에서 미정 값이 있어도 패턴이 우연히 매칭되는 문제가 생길 수 있다. 그래서 회사 코딩 가이드에 따라 `casex`를 금지하거나 `casez`만 허용하는 경우도 있다. 강의 취지대로 정리하면, 우선순위를 확실히 표현해야 할 때는 결국 `if-else`가 가장 안전하다.

### 도구 관점에서의 주의

Vivado Synthesis 문서(UG901)는 `case` 문 분기가 작성 순서대로 평가되고 첫 번째 참 분기가 실행된다고 설명한다. 동시에 priority processing을 피하고 싶으면 `parallel_case` 속성을 사용하라고 안내한다.

따라서 이 노트에서는 다음처럼 정리하는 것이 안전하다.

- 우선순위가 필요 없으면 상호배타적인 `case`로 쓴다.
- 우선순위가 필요하면 `if-else` 체인으로 명확하게 쓴다.
- 겹치는 `casez` / `casex` 패턴에 우선순위를 암묵적으로 기대하는 방식은 도구 의존성이 생길 수 있으므로 조심한다.

즉 "어떤 툴에서는 된다/안 된다"를 외우는 것보다, 우선순위가 필요한 회로는 아예 코드에서 우선순위를 분명히 드러내는 것이 더 중요하다.

## 5. Mux 예제

멀티플렉서(`multiplexer`)는 여러 입력 중 하나를 선택해서 출력하는 대표적인 조합회로다.

- 다수 개의 입력 중 하나를 선택한다.
- 선택 신호(`sel`)에 따라 출력이 결정된다.
- `if`, `case`, 조건 연산자 `?:` 등으로 모델링할 수 있다.

### 문법만 먼저 보기

```verilog
case (sel)
    2'b00: out = in0;
    2'b01: out = in1;
    ...
endcase
```

### 예제

```verilog
module mux_4x1 (
    input  [3:0] in0,
    input  [3:0] in1,
    input  [3:0] in2,
    input  [3:0] in3,
    input  [1:0] sel,
    output reg [3:0] out
);
    always @(*) begin
        case (sel)
            2'b00: out = in0;
            2'b01: out = in1;
            2'b10: out = in2;
            2'b11: out = in3;
            default: out = 4'b0000;
        endcase
    end
endmodule
```

### 해석

- Day 3 강의의 핵심 조합 블록과 동일한 구조다.
- 4개의 자릿수 중 하나를 고르는 FND 흐름과 직접 연결된다.

### 5.1 4비트 4:1 멀티플렉서 모델링 비교

수업에서는 `4비트 4:1 mux`를 여러 방식으로 쓸 수 있다는 점을 비교했다.

### `if` 문 사용

```verilog
module mux_4x1_if (
    input  [3:0] in0,
    input  [3:0] in1,
    input  [3:0] in2,
    input  [3:0] in3,
    input  [1:0] sel,
    output reg [3:0] out
);
    always @(*) begin
        if (sel == 2'b00)
            out = in0;
        else if (sel == 2'b01)
            out = in1;
        else if (sel == 2'b10)
            out = in2;
        else
            out = in3;
    end
endmodule
```

### `case` 문 사용

```verilog
module mux_4x1_case (
    input  [3:0] in0,
    input  [3:0] in1,
    input  [3:0] in2,
    input  [3:0] in3,
    input  [1:0] sel,
    output reg [3:0] out
);
    always @(*) begin
        case (sel)
            2'b00: out = in0;
            2'b01: out = in1;
            2'b10: out = in2;
            2'b11: out = in3;
            default: out = 4'b0000;
        endcase
    end
endmodule
```

### 조건 연산자 사용

```verilog
module mux_4x1_conditional (
    input  [3:0] in0,
    input  [3:0] in1,
    input  [3:0] in2,
    input  [3:0] in3,
    input  [1:0] sel,
    output [3:0] out
);
    assign out = (sel == 2'b00) ? in0 :
                 (sel == 2'b01) ? in1 :
                 (sel == 2'b10) ? in2 :
                                  in3;
endmodule
```

### 해석

- `if` 문은 우선순위가 있는 선택처럼 읽힌다.
- `case` 문은 선택지 나열 구조가 분명해서 mux 설명에 가장 자주 쓰인다.
- 조건 연산자 `?:`는 짧고 간결하지만, 깊어지면 가독성이 떨어질 수 있다.

즉 같은 `mux`라도 표현은 여러 가지가 가능하다. 교육용 설명과 보고서에는 `case`가 가장 무난하고, 간단한 식으로 줄이고 싶으면 조건 연산자도 사용할 수 있다.

### 5.2 4:2 이진 인코더 모델링

인코더(`encoder`)는 여러 입력 상태를 더 적은 비트수의 코드로 바꾸는 조합회로다. `4:2 binary encoder`는 보통 4개의 입력 중 하나가 활성화된 `one-hot` 입력을 받아, 그 위치를 2비트 코드로 출력한다고 이해하면 된다.

예를 들어 아래처럼 읽는다.

- `0001 -> 00`
- `0010 -> 01`
- `0100 -> 10`
- `1000 -> 11`

### `if` 문으로 쓰는 간단한 방식

입력 비트 수가 작을 때는 `if` 문으로 바로 써도 충분히 읽기 쉽다.

```verilog
module encoder_4to2_if (
    input  [3:0] in,
    output reg [1:0] out
);
    always @(*) begin
        out = 2'b00;

        if (in == 4'b0001)
            out = 2'b00;
        else if (in == 4'b0010)
            out = 2'b01;
        else if (in == 4'b0100)
            out = 2'b10;
        else if (in == 4'b1000)
            out = 2'b11;
    end
endmodule
```

### `for` 반복문과 비트 시프트를 이용한 방식

입력 비트 수가 커지면 같은 패턴을 반복해서 쓰는 것이 번거로워지므로, `for` 반복문을 쓰는 방식이 더 유리할 수 있다.

```verilog
module encoder_4to2_for (
    input  [3:0] in,
    output reg [1:0] out
);
    integer i;

    always @(*) begin
        out = 2'b00;

        for (i = 0; i < 4; i = i + 1) begin
            if (in == (4'b0001 << i))
                out = i[1:0];
        end
    end
endmodule
```

### 해석

- `4'b0001 << i` 는 `0001`, `0010`, `0100`, `1000`을 차례대로 만든다.
- 즉 `for` 문 안에서 "현재 입력이 one-hot 위치 `i`와 같은가"를 반복 검사하는 구조다.
- 조건이 맞으면 출력 `out`에 해당 인덱스 `i`를 2비트 코드로 넣는다.

### 왜 `for` 문이 유리한가

- 비트 수가 커질수록 같은 패턴을 반복 작성하기보다 반복문이 더 간결하다.
- 구조가 일정한 인코더, 디코더, 벡터 스캔 회로에서 재사용성이 좋다.
- 반대로 비트 수가 작으면 `if` 문으로 직접 나열해도 충분히 읽기 쉽다.

즉 작은 회로는 `if`로도 괜찮고, 입력 수가 늘어나면 `for` 반복문이 점점 더 유리해진다고 이해하면 된다.

## 6. Decoder와 LUT 예제

디코더(`decoder`)는 `m:n` 형태로 표현하는 경우가 많다. `m`비트 입력 코드를 받아서 `n`개의 출력선 중 하나를 활성화하는 구조다.

- `2:4 decoder`: 2비트 입력으로 4개 출력 중 하나 선택
- `3:8 decoder`: 3비트 입력으로 8개 출력 중 하나 선택
- 파라미터화된 디코더: 입력 비트 수와 출력 비트 수를 바꿔 재사용

인코더가 여러 입력을 짧은 코드로 "압축"한다면, 디코더는 코드를 다시 one-hot 성격의 출력으로 "풀어준다"고 보면 된다.

### 6.1 `m:n` 디코더와 2:4 디코더

`2:4 decoder`는 가장 기본적인 디코더 예제다.

| a[1:0] | y[3:0] |
| --- | --- |
| `00` | `0001` |
| `01` | `0010` |
| `10` | `0100` |
| `11` | `1000` |

### `if` 문으로 구현한 2:4 디코더

```verilog
module decoder_2to4_if (
    input  [1:0] a,
    output reg [3:0] y
);
    always @(*) begin
        y = 4'b0000;

        if (a == 2'b00)
            y = 4'b0001;
        else if (a == 2'b01)
            y = 4'b0010;
        else if (a == 2'b10)
            y = 4'b0100;
        else
            y = 4'b1000;
    end
endmodule
```

### `for` 반복문으로 구현한 2:4 디코더

```verilog
module decoder_2to4_for (
    input  [1:0] a,
    output reg [3:0] y
);
    integer i;

    always @(*) begin
        y = 4'b0000;

        for (i = 0; i < 4; i = i + 1) begin
            if (a == i[1:0])
                y[i] = 1'b1;
        end
    end
endmodule
```

### 해석

- 비트 수가 작을 때는 `if`로 직접 쓰는 편이 읽기 쉽다.
- 출력 수가 커질수록 같은 패턴 반복이 많아지므로 `for` 문이 더 유리하다.
- 결국 디코더는 "입력 값과 같은 인덱스의 출력 비트를 켠다"는 관점으로 이해하면 된다.

### 6.2 파라미터화된 디코더 모델링

수업에서는 파라미터화된 모듈 `dec_param`을 이용해서 입력/출력 크기를 바꿔가며 디코더를 재사용하는 감각도 함께 본다.

```verilog
module dec_param #(
    parameter N = 2,
    parameter M = 4
) (
    input  [N-1:0] a,
    output reg [M-1:0] y
);
    integer i;

    always @(*) begin
        y = {M{1'b0}};

        for (i = 0; i < M; i = i + 1) begin
            if (a == i)
                y[i] = 1'b1;
        end
    end
endmodule
```

### `dec_param`을 이용한 3:6 디코더

`3:6 decoder`는 3비트 입력 전체 경우의 수가 8개이지만, 여기서는 출력선을 6개만 두고 `0`부터 `5`까지만 유효하게 사용한다고 이해하면 된다. 입력이 `6`, `7`이면 모든 출력이 `0`으로 남는다.

```verilog
module decoder_3to6 (
    input  [2:0] a,
    output [5:0] y
);
    dec_param #(
        .N(3),
        .M(6)
    ) u_dec_param (
        .a(a),
        .y(y)
    );
endmodule
```

### 해석

- 파라미터화는 같은 구조를 입력/출력 크기만 바꿔 재사용하려는 목적이다.
- `dec_param`처럼 써두면 `2:4`, `3:6`, `3:8` 같은 변형을 쉽게 만들 수 있다.
- 교육용으로는 디코더 원리를 먼저 이해하고, 그 다음에 파라미터화로 일반화한다고 보면 된다.

### 6.3 BCD 디코더 예제

### 문법만 먼저 보기

```verilog
always @(*) begin
    case (bin)
        ...
        default: ...
    endcase
end
```

### 예제

```verilog
module bcd_to_7seg (
    input  [3:0] bin,
    output reg [7:0] seg
);
    always @(*) begin
        case (bin)
            4'd0: seg = 8'hc0;
            4'd1: seg = 8'hf9;
            4'd2: seg = 8'ha4;
            4'd3: seg = 8'hb0;
            default: seg = 8'hff;
        endcase
    end
endmodule
```

### 해석

- 입력 값에 따라 출력 패턴을 고정 매핑한다.
- 경우표를 코드로 옮긴 전형적인 조합회로다.

### 6.4 Truth table 기반 모델링: Binary-to-Gray 변환

수업에서는 진리표(`truth table`)를 그대로 `case` 문으로 옮기는 조합회로 예제로 `binary-to-gray` 코드 변환도 같이 봤다.

핵심 아이디어는 다음과 같다.

- `binary`는 수가 1씩 증가할 때 여러 비트가 한 번에 바뀔 수 있다.
- `gray`는 인접한 코드 사이에서 한 번에 한 비트만 바뀌도록 만든 코드다.

즉 `gray code`의 핵심은 "연속된 두 값 사이에서 단 한 비트만 변한다"는 점이다.

### 4비트 binary-to-gray truth table

| bin | gray |
| --- | --- |
| `0000` | `0000` |
| `0001` | `0001` |
| `0010` | `0011` |
| `0011` | `0010` |
| `0100` | `0110` |
| `0101` | `0111` |
| `0110` | `0101` |
| `0111` | `0100` |
| `1000` | `1100` |
| `1001` | `1101` |
| `1010` | `1111` |
| `1011` | `1110` |
| `1100` | `1010` |
| `1101` | `1011` |
| `1110` | `1001` |
| `1111` | `1000` |

### `case` 문을 이용한 모델링

```verilog
module bin_to_gray_case (
    input  [3:0] bin,
    output reg [3:0] gray
);
    always @(*) begin
        case (bin)
            4'b0000: gray = 4'b0000;
            4'b0001: gray = 4'b0001;
            4'b0010: gray = 4'b0011;
            4'b0011: gray = 4'b0010;
            4'b0100: gray = 4'b0110;
            4'b0101: gray = 4'b0111;
            4'b0110: gray = 4'b0101;
            4'b0111: gray = 4'b0100;
            4'b1000: gray = 4'b1100;
            4'b1001: gray = 4'b1101;
            4'b1010: gray = 4'b1111;
            4'b1011: gray = 4'b1110;
            4'b1100: gray = 4'b1010;
            4'b1101: gray = 4'b1011;
            4'b1110: gray = 4'b1001;
            4'b1111: gray = 4'b1000;
            default: gray = 4'b0000;
        endcase
    end
endmodule
```

### 해석

- 이 예제는 입력 `bin`에 대한 출력 `gray`를 진리표 그대로 옮긴 형태다.
- 진리표 기반 문제는 `case` 문이 가장 직관적일 때가 많다.
- 물론 나중에는 `gray = bin ^ (bin >> 1)`처럼 더 간단한 식으로도 쓸 수 있지만, 수업에서는 먼저 진리표와 코드 대응을 읽는 연습이 중요하다.

### 왜 Gray 코드를 알아야 하는가

강의에서 이 내용을 따로 강조한 이유는 나중에 asynchronous 설계와 FIFO에서 다시 나오기 때문이다.

- `binary` 카운터 값은 여러 비트가 동시에 바뀔 수 있다.
- clock domain crossing 상황에서는 여러 비트가 동시에 바뀌는 순간을 샘플링하면 중간값처럼 보일 위험이 있다.
- `gray` 코드는 인접 값 사이에서 한 비트만 바뀌므로, 이런 문제를 줄이는 데 유리하다.

그래서 `async FIFO`의 read/write pointer를 다룰 때 Gray 코드가 매우 자주 등장한다. 지금은 단순한 조합회로 변환처럼 보여도, 뒤에서 비동기 회로를 이해할 때 다시 중요해진다.

## 7. 4비트 비교기 모델링 비교

비교기(`comparator`)는 두 입력 값의 상대적인 크기를 비교하는 조합회로다.

- `a > b`
- `a == b`
- `a < b`

같은 비교 결과를 만든다. 수업에서는 `4비트 비교기`를 기준으로 `always`, `assign(관계 연산자)`, `for` 반복문 등으로 각각 구현한 형태를 비교한다고 보면 된다.

### `assign`과 관계 연산자로 구현

가장 직접적인 방식은 관계 연산자를 그대로 쓰는 것이다.

```verilog
module comparator_4bit_op (
    input  [3:0] a,
    input  [3:0] b,
    output gt,
    output eq,
    output lt
);
    assign gt = (a > b);
    assign eq = (a == b);
    assign lt = (a < b);
endmodule
```

### `always @(*)`와 `if` 조건문으로 구현

```verilog
module comparator_4bit_if (
    input  [3:0] a,
    input  [3:0] b,
    output reg gt,
    output reg eq,
    output reg lt
);
    always @(*) begin
        gt = 1'b0;
        eq = 1'b0;
        lt = 1'b0;

        if (a > b)
            gt = 1'b1;
        else if (a == b)
            eq = 1'b1;
        else
            lt = 1'b1;
    end
endmodule
```

### `for` 반복문으로 구현

비트 수가 커지면 최상위 비트부터 비교하는 구조를 반복문으로 표현할 수도 있다.

```verilog
module comparator_4bit_for (
    input  [3:0] a,
    input  [3:0] b,
    output reg gt,
    output reg eq,
    output reg lt
);
    integer i;

    always @(*) begin
        gt = 1'b0;
        eq = 1'b1;
        lt = 1'b0;

        for (i = 3; i >= 0; i = i - 1) begin
            if ((gt == 1'b0) && (lt == 1'b0) && (a[i] != b[i])) begin
                eq = 1'b0;

                if (a[i] > b[i])
                    gt = 1'b1;
                else
                    lt = 1'b1;
            end
        end
    end
endmodule
```

### 삼항연산자와 단순 연산자

삼항연산자 `?:`로도 비교 결과를 적을 수 있다.

```verilog
assign gt = (a > b) ? 1'b1 : 1'b0;
```

하지만 비교기에서는 위처럼 쓰기보다 아래처럼 관계 연산자를 바로 출력에 연결하는 편이 더 간단하다.

```verilog
assign gt = (a > b);
```

### 해석

- 비교기는 삼항연산자로도 쓸 수 있지만, 단순 비교 연산자만으로도 충분히 표현된다.
- 간단한 비교는 관계 연산자 기반 `assign`이 가장 짧고 읽기 쉽다.
- 여러 출력 플래그를 한 블록에서 명확히 제어하고 싶으면 `always @(*)`와 `if`가 좋다.
- 비트 수가 커지고 MSB부터 비교하는 구조를 강조하고 싶으면 `for` 반복문도 사용할 수 있다.

## 8. 3-State Bus 드라이버 모델링

3-state bus는 입력을 버스에 연결할지, 아니면 끊을지를 결정하는 구조다.

- enable이 `1`이면 신호원이 버스에 데이터를 실어 보낸다.
- enable이 `0`이면 신호원과 버스를 격리한다.
- 격리된 상태는 `high-impedance`, 즉 `Z` 상태로 표현한다.

여러 신호원이 하나의 공용 버스를 함께 사용할 때, 동시에 둘 이상이 버스를 강하게 구동하면 충돌이 나므로 보통 한 번에 하나만 enable되도록 제어한다.

### 데이터 버스 드라이버 그림 감각

```text
          en
          |
          v
data_in ------> [ 3-state bus driver ] ------> shared_bus
                                  when en = 1 : shared_bus = data_in
                                  when en = 0 : shared_bus = Z
```

이 그림에서 핵심은 "데이터를 보낸다"와 "버스에서 손을 뗀다"의 두 동작이다. 그래서 3-state는 값을 바꾸는 회로라기보다 연결 여부를 제어하는 회로로 이해하는 편이 좋다.

### 조건 연산자로 모델링

가장 직관적인 모델링은 조건 연산자로 enable 여부를 직접 쓰는 것이다.

```verilog
module tri_state_driver_cond (
    input       en,
    input [7:0] data_in,
    output [7:0] bus
);
    assign bus = en ? data_in : 8'bz;
endmodule
```

### 게이트 프리미티브 `bufif1`로 모델링

Verilog는 3-state 버퍼를 위한 게이트 프리미티브도 제공한다.

```verilog
module tri_state_driver_bufif1 (
    input       en,
    input [7:0] data_in,
    output [7:0] bus
);
    bufif1 (bus[0], data_in[0], en);
    bufif1 (bus[1], data_in[1], en);
    bufif1 (bus[2], data_in[2], en);
    bufif1 (bus[3], data_in[3], en);
    bufif1 (bus[4], data_in[4], en);
    bufif1 (bus[5], data_in[5], en);
    bufif1 (bus[6], data_in[6], en);
    bufif1 (bus[7], data_in[7], en);
endmodule
```

### 해석

- `bufif1`은 control 입력이 `1`일 때 입력을 출력으로 전달한다.
- control 입력이 `0`이면 출력은 `Z`가 된다.
- 동작 자체는 3-state 버퍼에 잘 맞지만, 코드가 길어지고 비트별로 써야 해서 교육용 예시가 아니라면 잘 쓰지 않는 경우도 많다.

### 게이트 프리미티브 사용 시 주의

- `bufif1` 같은 프리미티브는 문법적으로는 명확하지만, 타깃 디바이스와 합성 도구에 따라 내부 3-state를 그대로 구현하지 않고 다른 구조로 바꾸는 경우가 있다.
- 특히 FPGA 내부 로직에서는 진짜 내부 tri-state보다 mux 구조로 치환되는 경우가 많다.
- 그래서 RTL 수준에서는 조건 연산자나 `assign ... ? ... : 'bz` 형태가 더 읽기 쉽고 이식성도 좋다.
- 게이트 프리미티브는 교육용, 라이브러리 수준 모델링, 또는 I/O buffer 의미를 분명히 드러내고 싶을 때 제한적으로 보는 편이 안전하다.

즉 강의 맥락에서는 "3-state bus는 `Z`를 쓰는 조합회로"라는 감각을 익히고, 실제 RTL에서는 조건 연산자 방식이 더 실용적이라고 정리하면 된다.

### 8.1 8비트 양방향 버스 드라이버 모델링

양방향 버스 드라이버(`bidirectional bus driver`)는 한 신호선을 출력으로 구동할 수도 있고, 반대로 외부가 구동하는 값을 입력으로 읽을 수도 있는 구조다. 보통 `inout` 포트를 사용하고, 내가 버스를 구동하지 않을 때는 `Z`로 놓아 외부 장치가 버스를 사용할 수 있게 한다.

```verilog
module bidir_bus_driver (
    input        oe,
    input  [7:0] data_out,
    output [7:0] data_in,
    inout  [7:0] bidir_bus
);
    assign bidir_bus = oe ? data_out : 8'bz;
    assign data_in = bidir_bus;
endmodule
```

### 해석

- `oe = 1`이면 내부 `data_out`이 `bidir_bus`를 구동한다.
- `oe = 0`이면 내부는 버스에서 손을 떼고 `Z`를 낸다.
- 이때 외부가 버스를 구동하고 있으면 그 값을 `data_in`으로 읽을 수 있다.
- 수업에서 본 핵심은 `bidir_bus`를 `inout`으로 두고, `assign`으로 구동 여부를 정한다는 점이다.

실무에서는 내부 로직 전체를 양방향 버스로 설계하기보다, top-level I/O 근처에서만 `inout`을 쓰고 내부는 입력/출력 신호로 분리하는 경우가 많다.

## 9. 래치 방지 규칙

조합회로 설계에서 매우 중요하다.

### 형식만 먼저 보기

```verilog
always @(*) begin
    out = default;
    ...
end
```

### 해석

- 모든 분기에서 값을 할당해야 한다.
- 빠진 분기가 있으면 "이전 값을 유지하라"는 의미가 되어 래치가 생길 수 있다.
- 강의에서 `설계하지 않은 latch는 매우 위험하다`고 강조한 부분과 바로 연결된다.

### 9.1 원치 않는 래치의 합성

강의에서는 조합회로를 쓴다고 생각했는데도, 코드가 불완전하면 합성 결과에 래치가 들어갈 수 있다는 점을 예제로 보여준다. 여기서 말하는 "초기값"은 `initial` 문이 아니라, `always @(*)` 블록 맨 앞에서 주는 기본값(`default assignment`)이다.

### 초기값을 갖는 `case` 문 예제

```verilog
module case_with_default_value (
    input  [1:0] sel,
    input  [3:0] a,
    input  [3:0] b,
    input  [3:0] c,
    output reg [3:0] y
);
    always @(*) begin
        y = 4'b0000;

        case (sel)
            2'b00: y = a;
            2'b01: y = b;
            2'b10: y = c;
        endcase
    end
endmodule
```

### 초기값이 지정된 `if` 조건문 예제

```verilog
module if_with_default_value (
    input        en,
    input  [3:0] a,
    input  [3:0] b,
    output reg [3:0] y
);
    always @(*) begin
        y = 4'b0000;

        if (en)
            y = a & b;
    end
endmodule
```

### 해석

- 두 예제 모두 블록 시작 시 기본값을 한 번 주기 때문에, 어떤 조건이 오더라도 `y`는 항상 값이 정해진다.
- 그래서 합성 결과가 조합논리로 유지되고, 원치 않는 래치가 생기지 않는다.
- 수업에서 "초기값이 생겨서 래치가 없어진다"는 말은 바로 이 기본값 할당을 뜻한다.

### `if-else` 문이 래치를 포함한 회로로 합성될 수 있는 경우

`if-else`를 썼다고 해서 자동으로 안전해지는 것은 아니다. 조건을 다 덮지 못하면 래치가 추론될 수 있다.

```verilog
module latch_from_incomplete_ifelse (
    input  [1:0] sel,
    input  [3:0] a,
    input  [3:0] b,
    output reg [3:0] y
);
    always @(*) begin
        if (sel == 2'b00)
            y = a;
        else if (sel == 2'b01)
            y = b;
    end
endmodule
```

### 해석

- `sel == 2'b10`, `2'b11`일 때 `y`에 대한 대입이 없다.
- 그러면 합성기는 "그 경우에는 이전 값을 기억해야 한다"고 해석할 수 있다.
- 결과적으로 조합회로가 아니라 래치 포함 회로가 된다.

즉 `if-else`라는 문법 자체보다, 모든 경우에 출력이 할당되는지가 더 중요하다.

### 9.2 다중 `if` 문의 합성

강의에서 중요하게 보는 포인트는 `if`가 여러 개 연속으로 있을 때, 코드 순서가 실제 출력 우선순위에 영향을 준다는 점이다.

### 연속적인 `if` 조건문

```verilog
module multi_if_chain (
    input  a,
    input  b,
    input  c,
    output reg [1:0] y
);
    always @(*) begin
        y = 2'b00;

        if (a) y = 2'b01;
        if (b) y = 2'b10;
        if (c) y = 2'b11;
    end
endmodule
```

### 해석

- 이 구조에서는 `a`, `b`, `c`가 동시에 참일 수 있다.
- 그 경우 아래쪽 대입이 위쪽 대입을 덮어쓴다.
- 따라서 연속적인 `if` 문에서는 마지막 `if` 조건문이 가장 높은 우선순위를 갖는다.

### 다중 `if-else-if` 조건문

```verilog
module multi_if_else_if (
    input  a,
    input  b,
    input  c,
    output reg [1:0] y
);
    always @(*) begin
        y = 2'b00;

        if (a)
            y = 2'b01;
        else if (b)
            y = 2'b10;
        else if (c)
            y = 2'b11;
    end
endmodule
```

### 해석

- `if-else-if` 구조에서는 첫 번째로 참이 된 조건이 선택된다.
- 즉 위쪽 조건이 더 높은 우선순위를 갖는다.
- 연속 `if`와 반대로, 여기서는 맨 위 조건이 가장 먼저 판단되고 아래 조건은 앞 조건이 거짓일 때만 본다.

### 정리

- 연속 `if`: 아래쪽 `if`가 앞 대입을 덮어쓰므로 마지막 조건이 높은 우선순위를 갖는다.
- `if-else-if`: 위에서부터 순서대로 검사하므로 첫 번째 참 조건이 높은 우선순위를 갖는다.
- 여러 조건이 동시에 참일 수 있는지, 아니면 상호배타적인지를 먼저 생각하고 코드를 써야 한다.
- 조건 순서를 잘못 두면 의도와 다른 priority circuit가 합성될 수 있다.

슬라이드 마지막 줄은 정확히 못 적었더라도, 강의 맥락상 핵심은 "코드 순서가 우선순위 회로를 만든다"는 점으로 이해하면 된다.

## 10. 우선순위 인코더 감각

입력을 위에서부터 검사해야 하는 경우는 `if` 체인이 더 적절할 수 있다.

### 문법만 먼저 보기

```verilog
if      (in[3]) y = 2'd3;
else if (in[2]) y = 2'd2;
else if (in[1]) y = 2'd1;
else            y = 2'd0;
```

### 해석

- 조합회로라고 해서 모두 `case`가 정답은 아니다.
- 요구사항에 따라 우선순위를 명시해야 할 때가 있다.

## 자주 하는 실수

- 조합회로인데 sensitivity list를 부정확하게 쓴다.
- `default` 없이 `case`를 끝낸다.
- 조합회로 출력을 일부 분기에서만 할당한다.
- `assign`이 더 적절한 회로를 지나치게 장황한 `always`로 쓴다.
