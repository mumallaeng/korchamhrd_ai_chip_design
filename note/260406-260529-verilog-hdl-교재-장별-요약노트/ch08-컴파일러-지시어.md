## 이 장의 핵심

- 컴파일러 지시어는 회로 동작을 기술하는 문법이 아니라 `소스 해석과 전처리`를 제어하는 문법이다.
- 대표 지시어는 `` `timescale ``, `` `define ``, `` `undef ``, `` `include ``, `` `ifdef `` 계열이다.
- 지시어는 편리하지만 남용하면 코드 추적이 어려워진다.
- 실습 초반에는 `timescale`과 간단한 매크로 수준만 확실히 이해해도 충분하다.

8장은 Verilog 자체의 하드웨어 문법이 아니라, 소스 파일을 어떻게 읽고 확장할지를 제어하는 전처리 장이다.

## 현재 수업과 연결

- Day 1과 Day 2에서 계속 등장한 `` `timescale 1ns / 1ps ``가 가장 먼저 연결된다.
- 테스트벤치 상수, 공통 설정, 헤더 파일 분리를 체계화할 때 이 장의 지시어가 유용하다.

## 1. 지시어란 무엇인가

지시어는 백틱 `` ` `` 으로 시작한다.

- 회로 동작을 직접 만들지 않는다.
- 소스 코드가 읽히는 방식을 바꾼다.
- 전처리 단계에서 작동한다고 생각하면 된다.

### 문법만 먼저 보기

```verilog
`directive_name ...
```

### 예제

```verilog
`timescale 1ns / 1ps
`define WIDTH 8
```

### 해석

- `module`, `assign`, `always`와는 성격이 완전히 다르다.
- 하드웨어 구조보다 `소스 관리`에 가깝다.

## 2. `timescale`

초반에 가장 먼저 마주치는 지시어다.

### 문법만 먼저 보기

```verilog
`timescale 1ns / 1ps
```

### 시간 단위도 같이 보기

Verilog `timescale`에서 자주 쓰는 시간 단위 계열은 다음과 같다.

| 표기 | 의미 | 초 기준 |
| --- | --- | --- |
| `1s` | 1초 | $1 \text{ s}$ |
| `1ms` | 1밀리초 | $10^{-3} \text{ s}$ |
| `1us` | 1마이크로초 | $10^{-6} \text{ s}$ |
| `1ns` | 1나노초 | $10^{-9} \text{ s}$ |
| `1ps` | 1피코초 | $10^{-12} \text{ s}$ |
| `1fs` | 1펨토초 | $10^{-15} \text{ s}$ |

즉 현재 수업에서 자주 보는:

```verilog
`timescale 1ns / 1ps
```

는 다음 뜻으로 읽으면 된다.

| 항목 | 값 | 의미 |
| --- | --- | --- |
| 기본 시간 단위 | `1ns` | $10^{-9} \text{ s}$ |
| 시간 정밀도 | `1ps` | $10^{-12} \text{ s}$ |

### 어떻게 감각적으로 이해하면 좋은가

강의에서 말한 "기본 시간 단위는 사람이 보는 단위, 시간 정밀도는 시뮬레이터가 보는 단위"라는 설명은 입문 감각으로는 꽤 맞다.

조금 더 정확히 쓰면:

- 기본 시간 단위: 코드를 작성하는 사람이 `#5`, `#10` 같은 값을 어떤 크기로 읽을지 정하는 기준 스케일
- 시간 정밀도: 시뮬레이터가 시간을 내부적으로 자르고, 반올림하고, 이벤트를 배치하는 최소 해상도
- 사람 입장에서는 `time unit`이 "내가 지금 ns 기준으로 생각하고 있나, us 기준으로 생각하고 있나"를 잡아 준다.
- 시뮬레이터 입장에서는 `time precision`이 "여기서 어디까지를 같은 시점으로 보고, 어디서부터 다른 시점으로 나눌 것인가"를 정해 준다.

### 형식만 먼저 보기

```verilog
`timescale 1ns / 1ps
```

### 해석

- `#5`는 기본적으로 `5ns`로 읽힌다.
- 시뮬레이터는 시간을 `1ps` 수준까지 구분해서 처리한다.
- 따라서 `1ns / 1ps`는 "사람은 ns 스케일로 생각하고, 시뮬레이터는 ps 해상도까지 본다"라고 이해하면 된다.

### 작은 예시

| 설정 | `#5`의 의미 | 시뮬레이터 해상도 |
| --- | --- | --- |
| `` `timescale 1ns / 1ps `` | $5 \text{ ns}$ | $1 \text{ ps}$ |
| `` `timescale 1us / 1ns `` | $5 \text{ us}$ | $1 \text{ ns}$ |

즉 앞의 값은 "얼마나 큰 단위로 쓸 것인가", 뒤의 값은 "얼마나 잘게 구분할 것인가"에 가깝다.

### 예제

```verilog
`timescale 1ns / 1ps

module tb;
    reg clk;
    initial clk = 1'b0;
    always #5 clk = ~clk;
endmodule
```

### 해석

- 시뮬레이션 시간 단위와 정밀도를 지정한다.
- 테스트벤치의 `#5`, `#10` 같은 지연 해석과 직접 연결된다.
- 예를 들어 `` `timescale 1ns / 1ps ``에서 `#5`는 기본적으로 `5ns`로 읽힌다.
- 시간 정밀도가 더 작기 때문에, 시뮬레이터는 그보다 더 세밀한 시점까지 이벤트를 구분할 수 있다.

## 3. `define`

반복되는 상수나 간단한 표현을 매크로로 둘 수 있다.

### 문법만 먼저 보기

```verilog
`define NAME value
```

### 예제

```verilog
`define CLK_PERIOD 10

module tb;
    reg clk;

    initial clk = 1'b0;
    always #(`CLK_PERIOD/2) clk = ~clk;
endmodule
```

### 해석

- `CLK_PERIOD`라는 텍스트 치환 매크로다.
- 자주 바뀌는 상수를 한곳에 모으는 데 유용하다.
- 단, `parameter`와 달리 언어 차원의 타입/범위 구조는 아니다.

## 4. `undef`와 매크로 해제

흔히 "undefine"이라고 부르기도 하지만, 실제 Verilog 지시어 이름은 `` `undef``다.

### 문법만 먼저 보기

```verilog
`define NAME value
`undef NAME
```

### 예제

```verilog
`define DEBUG_LOG

module tb;
    initial begin
`ifdef DEBUG_LOG
        $display("debug enabled");
`endif

`undef DEBUG_LOG

`ifdef DEBUG_LOG
        $display("this line will not be compiled");
`endif

        $finish;
    end
endmodule
```

### 해석

- `` `undef``는 앞에서 정의한 매크로를 해제한다.
- 같은 파일 안에서 디버그 플래그 범위를 제한하고 싶을 때 유용하다.
- 매크로가 계속 살아 있어서 뒤쪽 코드에 예상치 못한 영향을 주는 것을 막을 수 있다.

## 5. `include`

외부 파일 내용을 현재 파일에 포함한다.

### 문법만 먼저 보기

```verilog
`include "defs.vh"
```

### 예제

```verilog
`include "tb_defs.vh"

module tb;
    reg clk;
    initial clk = TB_CLK_INIT;
endmodule
```

### 해석

- 공통 매크로, 공통 상수, 테스트벤치 설정 분리에 유용하다.
- 파일 의존성이 생기므로 경로 관리가 중요하다.

## 6. 조건부 컴파일

디버그용 코드나 환경별 코드를 분기할 수 있다.

### 문법만 먼저 보기

```verilog
`ifdef FLAG
    ...
`elsif OTHER_FLAG
    ...
`else
    ...
`endif
```

또는 "정의되지 않았을 때"를 보고 싶으면:

```verilog
`ifndef FLAG
    ...
`endif
```

### 지시어별 역할 짧게 보기

| 지시어 | 역할 |
| --- | --- |
| `` `ifdef`` | 매크로가 정의되어 있으면 아래 코드를 포함 |
| `` `ifndef`` | 매크로가 정의되어 있지 않으면 아래 코드를 포함 |
| `` `elsif`` | 앞 조건이 거짓일 때 다음 조건을 검사 |
| `` `else`` | 앞 조건들이 모두 거짓일 때 실행 |
| `` `endif`` | 조건부 컴파일 블록 종료 |

### 예제

```verilog
`define DEBUG_LOG

module tb;
    initial begin
`ifdef DEBUG_LOG
        $display("debug mode enabled");
`endif
        #10 $finish;
    end
endmodule
```

`ifndef`와 `elsif`를 같이 쓰면 이런 식이다.

```verilog
`define MODE_B

module tb_mode;
    initial begin
`ifdef MODE_A
        $display("mode A");
`elsif MODE_B
        $display("mode B");
`else
        $display("default mode");
`endif

`ifndef RELEASE
        $display("debug build");
`endif

        $finish;
    end
endmodule
```

### 해석

- 디버그 로그를 켜고 끄는 용도로 자주 쓸 수 있다.
- 설계 본체보다 테스트벤치/검증 코드에서 활용도가 높다.
- `` `ifdef``는 "있으면 포함", `` `ifndef``는 "없으면 포함"으로 읽으면 된다.
- `` `elsif``는 조건 분기를 길게 이어 붙일 때 유용하다.
- `` `endif``는 조건 블록을 닫는 마무리 지시어다.

## 7. `default_nettype`를 왜 알아두는가

초반부터 깊게 쓰지 않아도 좋지만, 실수를 줄이는 데 매우 도움이 된다.

### 문법만 먼저 보기

```verilog
`default_nettype none
```

### 해석

- 오타가 나면 암묵적 1비트 `wire`가 생기는 것을 막는다.
- 강의에서도 "자동 wire 선언은 오류를 부르기 쉽다"는 맥락이 있었는데, 그 문제를 더 엄격하게 잡는 방식이다.

## 8. `define`, `undef`, `parameter`를 같이 볼 때

- `` `define``: 매크로 생성
- `` `undef``: 매크로 해제
- `parameter`: 모듈 수준 파라미터

즉 `` `define``과 `` `undef``는 전처리 단계의 쌍이고, `parameter`는 설계 언어 자체의 재사용 수단이다.

## 9. `define`과 `parameter`의 차이

둘 다 상수를 다루지만 목적이 다르다.

| 항목 | `define` | `parameter` |
| --- | --- | --- |
| 단계 | 전처리 | 언어 수준 |
| 범위 | 텍스트 치환 | 모듈 파라미터 |
| 주 용도 | 공통 상수, 디버그 플래그 | 비트폭, 모듈 재사용 |

즉 설계 재사용에는 `parameter`, 파일 전역 설정에는 `define`이 더 자연스럽다.

## 자주 하는 실수

- 지시어와 Verilog 문법을 같은 층위로 본다.
- `define`을 남용해 코드 추적을 어렵게 만든다.
- `undef` 없이 매크로 범위를 길게 끌고 간다.
- `include` 경로를 엉성하게 관리한다.
- `parameter`로 해야 할 것을 무조건 매크로로 처리한다.
