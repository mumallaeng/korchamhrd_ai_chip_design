## 이 장의 핵심

- 시스템 태스크와 시스템 함수는 회로 자체를 만드는 문법이 아니라, `관찰`, `로그`, `제어`, `파일 처리`를 돕는 시뮬레이션 지원 도구다.
- 초반에는 크게 `디스플레이 계열`과 `파일 입출력 계열`로 나눠 이해하는 편이 가장 실용적이지만, 실제로는 `시뮬레이션 제어`, `시간`, `형변환`, `랜덤/분포`, `수학` 계열까지 넓게 존재한다.
- `$display`, `$write`, `$strobe`, `$monitor`의 차이와, `$fopen`, `$fclose`, `$fdisplay`, `$fwrite`, `$readmemh`의 역할을 잡아 두면 테스트벤치 생산성이 크게 올라간다.
- 특히 `$display`는 자동 줄바꿈이 있고, `$write`는 자동 줄바꿈이 없다는 차이를 기억해 두면 로그를 읽기 훨씬 쉬워진다.
- 일부 시스템 함수는 도구나 언어 버전에 따라 지원 범위 차이가 있으므로, 실제 사용 전 시뮬레이터 지원 여부를 확인하는 습관이 필요하다.

## 1. 시스템 태스크와 시스템 함수란 무엇인가

둘 다 이름 앞에 `$`가 붙는다.

- 시스템 태스크: 보통 반환값 없이 동작 수행
- 시스템 함수: 값을 반환해서 식 안에서 사용 가능

### 문법만 먼저 보기

```verilog
$display("...");
$finish;
value = $time;
```

### 해석

- `$display`, `$finish`는 시스템 태스크다.
- `$time`은 시스템 함수다.
- 목적은 회로 생성이 아니라 시뮬레이션 관찰과 제어다.

## 3. 디스플레이 계열 시스템 태스크

콘솔에 메시지를 찍어 상태를 확인하는 계열이다.  
테스트벤치에서 가장 먼저 익히는 그룹이기도 하다.

### 3.1 어떤 종류가 있는가

| 태스크                             | 핵심 역할           | 메모                            |
| ------------------------------- | --------------- | ----------------------------- |
| `$display`                      | 한 번 출력          | 자동 줄바꿈 있음, 현재 실행 시점 값을 바로 봄   |
| `$displayb`                     | 한 번 출력          | 기본 표시를 binary 중심으로 봄          |
| `$displayh`                     | 한 번 출력          | 기본 표시를 hex 중심으로 봄             |
| `$displayo`                     | 한 번 출력          | 기본 표시를 octal 중심으로 봄           |
| `$write`                        | 한 번 출력          | 자동 줄바꿈 없음                     |
| `$writeb`                       | 한 번 출력          | binary 계열, 줄바꿈 없음             |
| `$writeh`                       | 한 번 출력          | hex 계열, 줄바꿈 없음                |
| `$writeo`                       | 한 번 출력          | octal 계열, 줄바꿈 없음              |
| `$strobe`                       | 한 번 출력          | 현재 time slot 끝에서 결과가 반영된 뒤 출력 |
| `$strobeb/$strobeh/$strobeo`    | 한 번 출력          | radix 변형                      |
| `$monitor`                      | 값이 바뀔 때마다 계속 출력 | 지속 관찰용                        |
| `$monitorb/$monitorh/$monitoro` | 값이 바뀔 때마다 계속 출력 | radix 변형                      |
| `$monitoron`                    | monitor 출력 다시 켬 | monitor 제어                    |
| `$monitoroff`                   | monitor 출력 끔    | monitor 제어                    |

초반에는 아래만 확실히 익혀도 충분하다.

- `$display`
- `$write`
- `$strobe`
- `$monitor`
- `$monitoron`
- `$monitoroff`

### 3.2 `$display`

가장 기본적인 출력 태스크다.

### 문법만 먼저 보기

```verilog
$display("a=%b b=%b y=%b", a, b, y);
```

### 예제

```verilog
initial begin
    $display("start simulation");
    $display("a=%b b=%b y=%b", a, b, y);
end
```

### 해석

- 그 시점 값을 한 번 출력한다.
- 자동 줄바꿈이 있다.
- 가장 읽기 쉬운 기본 로그 방식이다.
- 시뮬레이터 관점으로 보면, 현재 문장이 실행되는 순간의 값을 바로 찍는다고 이해하면 된다.
- 그래서 같은 time slot 안에서 뒤이어 반영될 논블로킹 할당 결과까지 기다리지는 않는다.
- 블로킹 대입 `=`처럼 문장 순서대로 값이 바로 반영되는 흐름을 따라갈 때는 `$display`가 특히 직관적이다.

## 3.3 `$write`

`$display`와 거의 비슷하지만 자동 줄바꿈이 없다.

### 문법만 먼저 보기

```verilog
$write("a=%b ", a);
$write("b=%b ", b);
$write("y=%b", y);
```

### 예제

```verilog
initial begin
    $write("a=%b ", a);
    $write("b=%b ", b);
    $write("y=%b\n", y);
end
```

### 해석

- 줄을 이어서 직접 만들고 싶을 때 유용하다.
- 자동 줄바꿈이 없으므로 마지막에 `\n`을 넣을지 직접 결정해야 한다.
- `$display`보다 제어력이 높지만, 남용하면 로그가 지저분해질 수 있다.

## 3.4 `$strobe`

현재 time slot이 끝나고 값이 안정된 뒤 출력한다.

### 문법만 먼저 보기

```verilog
$strobe("q=%b", q);
```

### 예제

```verilog
always @(posedge clk) begin
    q <= d;
    $display("display q=%b", q);
    $strobe("strobe  q=%b", q);
end
```

### 해석

- 시뮬레이터 관점으로 보면, `$display`는 현재 time slot에서 "처음 실행되는 그 시점"의 값을 본다.
- `$strobe`는 같은 time slot 안의 업데이트가 끝난 뒤, 즉 마지막 결과값이 반영된 후의 값을 본다.
- 그래서 `$strobe`는 "결과가 settle된 뒤에 출력한다"는 표현으로 이해하면 된다.
- 논블로킹 할당 `<=`가 있을 때 차이를 보기 좋다.
- 특히 논블로킹 대입 `<=`의 결과값을 보고 싶을 때는 `$strobe` 쪽이 더 권장된다.

### 3.4.1 왜 둘이 다르게 보이는가

시뮬레이터는 한 시각에서도 내부적으로 여러 단계로 이벤트를 처리한다.

- `$display`는 현재 실행 중인 문맥에서 바로 출력한다.
- 논블로킹 할당 `<=`의 실제 갱신은 같은 time slot의 뒤쪽 단계에서 반영될 수 있다.
- `$strobe`는 그 뒤쪽 단계까지 지난 다음 출력하므로, 최종 반영된 값을 보기 쉽다.

따라서 "같은 시간인데 왜 출력값이 다르지"라는 느낌이 들면, 시간 자체가 다른 것이 아니라 같은 time slot 안에서 관찰 시점이 다른 것이라고 보면 된다.

### 3.4.2 언제 `$display`를 쓰고 언제 `$strobe`를 쓰는가

실전에서는 아래처럼 구분하면 편하다.

- 블로킹 대입 `=`이나 순차적으로 실행되는 계산 흐름을 따라가고 싶을 때는 `$display`
- 논블로킹 대입 `<=`의 최종 반영 결과를 보고 싶을 때는 `$strobe`

즉, `$display`는 "지금 이 줄이 실행될 때 무슨 값이 보이냐"에 가깝고, `$strobe`는 "이 time slot이 끝났을 때 최종적으로 무슨 값이 됐냐"에 가깝다.

다만 여기서 말하는 "순서대로 실행된다"는 느낌은 절차적 블로킹 대입 문맥을 두고 하는 말에 가깝다. `assign` 키워드의 연속 할당과는 구분해서 보는 편이 안전하다.

## 3.5 `$monitor`

등록한 인자 값이 바뀔 때마다 자동으로 계속 출력한다.

### 문법만 먼저 보기

```verilog
$monitor("t=%0t state=%b out=%b", $time, state, out);
```

### 예제

```verilog
initial begin
    $monitor("t=%0t a=%b b=%b y=%b", $time, a, b, y);
end
```

### 해석

- 카운터, FSM, 상태 신호처럼 "변화 추적"이 중요한 회로에 특히 좋다.
- 파형 창 없이도 동작 흐름을 빠르게 확인할 수 있다.
- 한 번 `$monitor(...)`를 걸어 두면, 지정한 신호 중 하나라도 바뀔 때마다 자동으로 출력하므로 계속 다시 부를 필요가 없다.
- 반복해서 `$display`를 넣기 귀찮은 구간에서는 `$monitor`가 더 편하다.
- 새 `$monitor`를 다시 걸면 기존 monitor를 대체한다고 이해하면 된다.
- 너무 많은 신호를 걸면 로그가 과해질 수 있다.

## 3.6 `$monitoron` / `$monitoroff`

monitor 출력 자체를 켜고 끌 수 있다.

### 문법만 먼저 보기

```verilog
$monitor(...);
$monitoroff;
$monitoron;
```

### 예제

```verilog
initial begin
    $monitor("t=%0t q=%b", $time, q);
    #50 $monitoroff;
    #20 $monitoron;
end
```

### 해석

- 관심 구간에서만 로그를 보고 싶을 때 유용하다.
- 긴 시뮬레이션에서 로그 양을 줄이는 데 도움이 된다.

## 3.7 radix 변형 계열은 어떻게 보면 되는가

`$displayb`, `$displayh`, `$displayo` 같은 이름은 "같은 계열인데 기본 표시 진법을 다르게 보는 변형" 정도로 이해하면 된다.

### 형식만 먼저 보기

```verilog
$displayb(expr);
$displayh(expr);
$displayo(expr);
```

### 해석

- 초반에는 포맷 문자열 `%b`, `%h`, `%o`를 사용하는 방식이 더 읽기 쉽다.
- 변형 이름 자체는 "이런 계열이 있다"는 정도를 먼저 기억하면 충분하다.

## 3.8 자주 보는 출력 형식 지정 문자

`$display`, `$write`, `$strobe`, `$monitor`, `$fdisplay`, `$fwrite` 등은 같은 형식 지정 문자를 공유한다.

### 형식만 먼저 보기

```verilog
$display("bin=%b hex=%h dec=%d", sig, sig, sig);
$display("char=%c str=%s time=%t", ch, msg, $realtime);
$display("inst=%m");
```

### 표로 먼저 보기

| 형식 | 의미 | 메모 |
| --- | --- | --- |
| `%h`, `%H` | hexadecimal | 16진수 출력 |
| `%d`, `%D` | decimal | 10진수 출력 |
| `%o`, `%O` | octal | 8진수 출력 |
| `%b`, `%B` | binary | 2진수 출력 |
| `%c`, `%C` | character | 보통 하위 8비트를 문자로 봄 |
| `%s`, `%S` | string | 문자열 출력 |
| `%t`, `%T` | time | 보통 `$time`, `$realtime`와 함께 쓰고 `$timeformat` 영향을 받음 |
| `%m`, `%M` | hierarchical name | 인자 없이 현재 인스턴스/계층 이름을 찍을 때 유용 |
| `%v`, `%V` | value + strength | strength까지 같이 보고 싶을 때 |
| `%l`, `%L` | library binding information | 덜 자주 보이고, 도구/버전 차이를 의식하는 편이 안전 |
| `%u`, `%U` | unformatted 2-value data | 보통 `0`, `1` 중심으로 보는 쪽, 덜 자주 사용 |
| `%z`, `%Z` | unformatted 4-value data | `0`, `1`, `x`, `z`를 가공 없이 보고 싶을 때 |

### 해석

- 초반에는 `%b`, `%h`, `%d`, `%m`, `%t`만 익혀도 대부분의 로그는 충분히 읽을 수 있다.
- `%m`은 현재 인스턴스 경로를 보여 주므로, 같은 모듈이 여러 번 인스턴스화된 설계에서 특히 유용하다.
- `%t`는 시간 표시용이므로 `$timeformat`과 같이 보는 습관이 좋다.
- `%v`, `%l`, `%u`, `%z`는 상대적으로 덜 자주 보며, 도구 지원 차이나 언어 버전 차이를 확인하는 편이 안전하다.

### 예제

```verilog
initial begin
    $timeformat(-9, 0, " ns", 8);
    $display("inst=%m time=%t a_bin=%b a_hex=%h a_dec=%d", $realtime, a, a, a);
end
```

### 해석

- `%m`으로 어느 인스턴스에서 찍은 로그인지 알 수 있다.
- `%t`로 현재 시간을 통일된 형식으로 볼 수 있다.
- 같은 신호라도 `%b`, `%h`, `%d`를 같이 놓으면 해석이 훨씬 빨라진다.

## 4. 파일 입출력 계열 시스템 태스크

콘솔이 아니라 파일에 로그를 남기거나, 파일에서 데이터를 읽는 계열이다.

### 4.1 어떤 종류가 있는가

| 태스크 | 핵심 역할 | 메모 |
| --- | --- | --- |
| `$fopen` | 파일 열기 | 파일 핸들 반환 |
| `$fclose` | 파일 닫기 | 열린 파일 정리 |
| `$fdisplay` | 파일에 한 줄 출력 | `$display`의 파일 버전, 자동 줄바꿈 |
| `$fwrite` | 파일에 출력 | `$write`의 파일 버전, 자동 줄바꿈 없음 |
| `$fgets` | 한 줄 읽기 | 텍스트 라인 입력 |
| `$fstrobe` | 파일에 안정 후 출력 | `$strobe`의 파일 버전 |
| `$fmonitor` | 파일에 변화 추적 출력 | `$monitor`의 파일 버전 |
| `$fscanf` | 파일에서 formatted read | 입력 파싱 |
| `$fread` | binary/block 읽기 | 바이너리 데이터 입력 |
| `$fseek` | 파일 위치 이동 | 파일 포인터 제어 |
| `$rewind` | 파일 처음으로 되돌림 | 보통 `$fseek(fd, 0, 0)`와 같은 의미로 봄 |
| `$ftell` | 현재 파일 위치 반환 | offset 확인 |
| `$fflush` | 버퍼 flush | buffered output 강제 기록 |
| `$feof` | EOF 상태 확인 | end-of-file 체크 |
| `$ferror` | 파일 오류 상태 확인 | 에러 확인 |
| `$ungetc` | 문자 되돌려 넣기 | 입력 스트림 제어 |
| `$readmemh` | hex 메모리 파일 읽기 | ROM / vector 로드 |
| `$readmemb` | binary 메모리 파일 읽기 | ROM / vector 로드 |

초반에는 아래만 우선 익히면 된다.

- `$fopen`
- `$fclose`
- `$fdisplay`
- `$fwrite`
- `$fgets`
- `$readmemh`
- `$readmemb`

문자열 포맷/파싱 계열도 같이 알아두면 편하다.

| 항목 | 역할 | 메모 |
| --- | --- | --- |
| `$swrite` | 문자열 버퍼에 쓰기 | `$write`의 문자열 버전처럼 이해 가능 |
| `$sformat` | format string 해석 후 문자열 생성 | 포맷 기반 문자열 작성 |
| `$sscanf` | 문자열에서 formatted read | 이미 읽은 문자열 파싱 |

## 4.2 `$fopen` / `$fclose`

파일 입출력의 시작과 끝이다.

### 문법만 먼저 보기

```verilog
fd = $fopen("tb_log.txt", "w");
$fclose(fd);
```

### 예제

```verilog
integer fd;

initial begin
    fd = $fopen("tb_log.txt", "w");
    if (fd == 0) begin
        $display("failed to open file");
        $finish;
    end

    $fclose(fd);
end
```

### 해석

- `fd`는 파일 핸들이다.
- 열기에 실패하면 `0`을 반환하는 경우가 많으므로 바로 검사하는 습관이 좋다.
- 열었으면 닫는 습관까지 같이 가져가야 한다.

## 4.3 `$fdisplay`와 `$fwrite`

콘솔의 `$display`, `$write`와 거의 같은 관계다.

### 문법만 먼저 보기

```verilog
$fdisplay(fd, "a=%0d b=%0d", a, b);
$fwrite(fd, "sum=%0d", sum);
```

### 예제

```verilog
integer fd;

initial begin
    fd = $fopen("result.log", "w");

    $fdisplay(fd, "start");
    $fwrite(fd, "a=%0d ", a);
    $fwrite(fd, "b=%0d ", b);
    $fwrite(fd, "sum=%0d\n", sum);

    $fclose(fd);
end
```

### 해석

- `$fdisplay`는 자동 줄바꿈이 있다.
- `$fwrite`는 자동 줄바꿈이 없다.
- 즉 `$fdisplay`는 "한 줄 로그", `$fwrite`는 "내가 줄 형식을 직접 짜는 로그"로 이해하면 편하다.

## 4.4 `$fscanf`

파일에서 formatted read를 할 때 쓴다.

### 문법만 먼저 보기

```verilog
code = $fscanf(fd, "%d %d", a, b);
```

### 예제

```verilog
integer fd;
integer code;
integer a_in, b_in;

initial begin
    fd = $fopen("input.txt", "r");
    code = $fscanf(fd, "%d %d", a_in, b_in);
    $display("read a=%0d b=%0d code=%0d", a_in, b_in, code);
    $fclose(fd);
end
```

### 해석

- 테스트 벡터를 텍스트 파일에서 읽어 올 때 유용하다.
- 읽은 개수나 성공 여부를 함께 확인하는 습관이 좋다.

## 4.5 `$fgets`, `$fread`, `$sscanf`

텍스트 라인 단위로 읽을지, 바이너리 블록처럼 읽을지에 따라 도구가 달라진다.

### 문법만 먼저 보기

```verilog
code = $fgets(str, fd);
code = $fread(mem, fd);
code = $sscanf(str, "%d %d", a, b);
```

### 예제

```verilog
integer fd;
integer code;
reg [8*64-1:0] line_buf;
integer a_in, b_in;

initial begin
    fd = $fopen("input.txt", "r");
    code = $fgets(line_buf, fd);
    code = $sscanf(line_buf, "%d %d", a_in, b_in);
    $display("line parsed a=%0d b=%0d code=%0d", a_in, b_in, code);
    $fclose(fd);
end
```

### 해석

- `$fgets`는 파일에서 한 줄을 읽는 데 유용하다.
- `$sscanf`는 이미 읽어 온 문자열을 다시 포맷 기반으로 파싱한다.
- `$fread`는 바이너리/블록 형태 입력을 읽을 때 적합하다.
- 즉 텍스트 기반 테스트 벡터라면 `$fgets + $sscanf`, 메모리 블록이라면 `$fread`를 먼저 떠올리면 된다.

## 4.6 `$swrite`, `$sformat`

문자열을 "만드는" 쪽에서 자주 보는 이름들이다.

### 문법만 먼저 보기

```verilog
$swrite(str, "a=%0d b=%0d", a, b);
$sformat(str, "sum=%0d", sum);
```

### 예제

```verilog
reg [8*64-1:0] line_buf;

initial begin
    $swrite(line_buf, "a=%0d b=%0d", a, b);
    $display("buf=%0s", line_buf);

    $sformat(line_buf, "sum=%0d", sum);
    $display("buf=%0s", line_buf);
end
```

### 해석

- `$swrite`는 문자열 버퍼에 기록하는 쪽으로 이해하면 된다.
- `$sformat`은 format string을 해석해서 문자열 버퍼를 만든다.
- 로그를 바로 찍는 대신 "문자열을 먼저 만들고 나중에 찍거나 파일에 쓰는" 흐름을 만들 때 유용하다.

## 4.7 `$readmemh` / `$readmemb`

메모리 초기화 파일을 한 번에 읽어 온다.

### 문법만 먼저 보기

```verilog
$readmemh("rom.hex", mem);
$readmemb("rom.bin", mem);
```

### 예제

```verilog
reg [7:0] mem [0:15];

initial begin
    $readmemh("rom.hex", mem);
    $display("mem[0]=%h mem[1]=%h", mem[0], mem[1]);
end
```

### 해석

- ROM 초기값, LUT 데이터, 테스트 벡터 로드에 매우 유용하다.
- 나중에 메모리 기반 설계를 할 때 특히 중요해진다.

## 4.8 `$fseek`, `$rewind`, `$ftell`, `$fflush`, `$ferror`, `$feof`, `$ungetc`

초반부터 자주 쓰지는 않지만, 파일 제어를 조금 더 정밀하게 다룰 때 도움이 된다.

### 문법만 먼저 보기

```verilog
code = $fseek(fd, offset, origin);
code = $rewind(fd);
pos  = $ftell(fd);
code = $fflush(fd);
code = $ferror(fd, errstr);
code = $feof(fd);
code = $ungetc(ch, fd);
```

### 해석

- `$fseek`: 파일 포인터를 이동한다.
- `$rewind`: 파일 포인터를 처음으로 되돌린다.
- `$ftell`: 현재 파일 위치를 알려 준다.
- `$fflush`: buffered output을 즉시 밀어 넣는다.
- `$ferror`: 파일 관련 오류 상태를 확인한다.
- `$feof`: end-of-file 상태를 확인한다.
- `$ungetc`: 읽은 문자를 다시 입력 스트림에 되돌려 넣는다.

즉 이들은 "파일을 열고 읽고 닫는 기본 흐름"보다 더 세밀한 제어가 필요할 때 쓰는 도구들이다.

## 4.9 `$sdf_annotate`

이 장의 초반 우선순위는 아니지만, 종류를 볼 때는 꼭 이름을 알아둘 가치가 있다.

### 문법만 먼저 보기

```verilog
$sdf_annotate("design.sdf", dut);
```

### 해석

- SDF(Standard Delay Format) 파일의 지연 정보를 설계 인스턴스에 back-annotation 하는 태스크다.
- 보통 gate-level simulation이나 타이밍 검증 문맥에서 본다.
- 현재 수업 단계에서는 "고급 타이밍 검증용 시스템 태스크" 정도로 이해하면 충분하다.

## 5. 시뮬레이션 제어 / 시간 / 타임스케일 계열

디스플레이와 파일 I/O 말고도 실제로 매우 자주 보게 되는 그룹이다.

### 5.1 어떤 종류가 있는가

| 항목 | 역할 | 메모 |
| --- | --- | --- |
| `$finish` | 시뮬레이션 종료 | 가장 자주 사용 |
| `$stop` | 시뮬레이션 일시 중지 | 디버깅 중간 정지 |
| `$printtimescale` | 설계의 timescale 표시 | time unit/precision 확인 |
| `$timeformat` | 시간 출력 형식 설정 | `%t` 출력 형식 제어 |
| `$time` | 현재 시뮬레이션 시간 | 정수 시간 |
| `$stime` | 현재 시뮬레이션 시간 | 짧은 정수 시간 표현 |
| `$realtime` | 현재 시뮬레이션 시간 | real 기반 시간 값 |

### 문법만 먼저 보기

```verilog
$stop;
$finish;
$printtimescale(dut);
$timeformat(-9, 0, " ns", 10);
t0 = $time;
rt = $realtime;
```

### 예제

```verilog
initial begin
    $timeformat(-9, 0, " ns", 10);
    $display("time=%0t", $time);
    $display("stime=%0d", $stime);
    $display("realtime=%0f", $realtime);
    #20;
    $finish;
end
```

### 해석

- `$finish`와 `$stop`은 시뮬레이션 제어의 핵심이다.
- `$time`, `$stime`, `$realtime`은 현재 시간을 서로 다른 형태로 본다.
- `$timeformat`은 `%t` 출력 형식을 사람이 읽기 좋게 정리할 때 유용하다.
- `$printtimescale`은 모듈이 어떤 time unit / precision으로 해석되는지 확인하는 데 도움된다.

## 6. 자료형 변환 계열

테스트벤치와 보조 계산 코드에서는 타입 해석을 바꿔야 할 때가 자주 있다.

### 6.1 어떤 종류가 있는가

| 항목 | 역할 | 메모 |
| --- | --- | --- |
| `$itor` | integer -> real | 정수를 실수로 변환 |
| `$rtoi` | real -> integer | 실수를 정수로 변환 |
| `$bitstoreal` | bit vector -> real | 비트 패턴을 real로 해석 |
| `$realtobits` | real -> bit vector | real 값을 비트 패턴으로 변환 |
| `$signed` | signed 해석 | 부호 있는 값으로 보기 |
| `$unsigned` | unsigned 해석 | 부호 없는 값으로 보기 |

### 문법만 먼저 보기

```verilog
r0   = $itor(i0);
i1   = $rtoi(r1);
bits = $realtobits(r2);
r3   = $bitstoreal(bits);
s0   = $signed(raw);
u0   = $unsigned(raw);
```

### 예제

```verilog
integer i0;
real    r0;
reg [31:0] raw;

initial begin
    i0 = 25;
    r0 = $itor(i0);
    $display("itor=%0f", r0);

    raw = 32'hffff_ff80;
    $display("signed=%0d unsigned=%0d", $signed(raw), $unsigned(raw));
end
```

### 해석

- `$itor`, `$rtoi`는 실수 계산 보조에서 자주 쓴다.
- `$signed`, `$unsigned`는 같은 비트 패턴을 다른 해석으로 읽고 싶을 때 중요하다.
- `$bitstoreal`, `$realtobits`는 더 저수준의 real-bit 변환이다.

## 7. 랜덤 / 분포 / 수학 계열

검증 입력을 다양하게 만들거나, real 기반 계산을 보조할 때 쓰는 그룹이다.

### 7.1 어떤 종류가 있는가

| 큰 분류 | 대표 항목 | 메모 |
| --- | --- | --- |
| 기본 난수 | `$random` | 가장 기본적인 pseudo-random |
| 분포 계열 | `$dist_uniform`, `$dist_normal`, `$dist_exponential`, `$dist_poisson` | 분포 기반 난수 |
| 비트 수 계산 계열 | `$clog2` | 필요한 폭을 정할 때 자주 사용 |
| 수학 계열 | `$sqrt`, `$pow`, `$exp`, `$ln`, `$log10`, `$sin`, `$cos`, `$atan2` | 실수 계산 보조 |

### 문법만 먼저 보기

```verilog
rnd = $random(seed);
u   = $dist_uniform(seed, 0, 255);
n   = $dist_normal(seed, mean, sigma);
aw  = $clog2(depth);
x   = $sqrt(val);
y   = $pow(base, exp);
w   = $exp(val);
z   = $atan2(y0, x0);
```

### 7.2 `$random`은 왜 쓰는가

강의에서 말한 핵심은 "패터닝된 검증만 하지 않기 위해서"에 가깝다.

- 입력을 항상 같은 순서로만 주면, 예를 들어 상위 비트부터 차례대로 훑는 방식에서는 특정 조합 버그가 늦게 드러나거나 아예 안 보일 수 있다.
- `$random`을 쓰면 입력 순서가 고정되지 않아서, 사람이 미처 생각하지 못한 조합을 더 빨리 건드릴 수 있다.
- 그래서 전체 결론이 비슷하더라도, 초반 코드 안정화 단계에서는 랜덤 검증이 시간을 줄여 주는 경우가 많다.
- 다만 corner case는 성격이 다르다. 최대값, 최소값, all-zero, all-one, carry 전파, overflow처럼 "반드시 확인해야 하는 경계조건"은 랜덤만으로는 놓칠 수 있다.
- 따라서 실제 검증은 `directed test + random test`를 같이 가져가는 편이 가장 실용적이다.

### 형식만 먼저 보기

```verilog
seed = 32'h1234;

repeat (20) begin
    a = $random(seed);
    b = $random(seed);
    #10;
end
```

### 예제 1: 기본 사용

```verilog
integer seed;
integer rnd_v;
real    root_v;

initial begin
    seed = 32'h1234;
    rnd_v = $random(seed);
    root_v = $sqrt(16.0);

    $display("random=%0d", rnd_v);
    $display("sqrt=%0f", root_v);
end
```

### 해석

- `$random`은 패턴 테스트를 전부 손으로 나열하지 않고도 다양한 입력을 빨리 흘려 넣는 데 유용하다.
- 입력을 상위 비트부터 고정 순서로만 주면 특정 버그가 드러나지 않을 수 있는데, 랜덤 입력은 그런 누락을 줄이는 데 도움이 된다.
- 검증 과정이 짧아지고, 비슷한 결론에 더 빨리 도달할 때가 많아 코드 안정화 속도가 올라간다.
- 다만 코너 케이스는 랜덤만으로 충분하지 않을 수 있으므로, 경계조건 테스트는 별도로 가져가는 편이 안전하다.
- 분포 계열과 수학 계열은 도구 지원 범위 차이가 있을 수 있으므로 실제 사용 전 확인하는 습관이 좋다.

### 예제 2: directed test와 random test를 같이 쓰기

```verilog
integer seed;
integer i;
reg [3:0] a, b;
wire [4:0] sum;

assign sum = a + b;

initial begin
    seed = 32'h2026;

    // 1) directed test: 꼭 확인해야 하는 경계조건
    a = 4'h0; b = 4'h0; #10;
    a = 4'hf; b = 4'h0; #10;
    a = 4'hf; b = 4'hf; #10;
    a = 4'h7; b = 4'h8; #10;

    // 2) random test: 패턴화되지 않은 다양한 입력
    for (i = 0; i < 20; i = i + 1) begin
        a = $random(seed);
        b = $random(seed);
        #10;
        $display("RND i=%0d a=%0d b=%0d sum=%0d", i, a, b, sum);
    end
end
```

### 해석

- 앞부분 directed test는 "반드시 지나가야 하는 케이스"를 고정 확인한다.
- 뒷부분 random test는 사람이 순서를 설계하지 않아도 다양한 조합을 빠르게 흘려 보낸다.
- 실무 감각으로는 random이 초반 버그 탐지 속도를 올려 주고, directed가 마지막 신뢰성을 닫아 주는 역할을 한다.

### 7.3 `$clog2`는 언제 쓰는가

`$clog2`는 "몇 비트가 필요하냐"를 계산할 때 가장 자주 보는 함수다.

- 의미는 `2^n >= value`를 만족하는 가장 작은 정수 `n`을 구하는 쪽에 가깝다.
- 그래서 메모리 주소 폭, 카운터 폭, 인덱스 폭을 정할 때 자주 쓴다.
- 예를 들어 깊이가 256이면 8비트가 필요하고, 깊이가 10이면 4비트가 필요하다.
- 강의나 책이 Verilog HDL 문맥이어도, 실제 실습 도구에서는 `$clog2`를 자주 지원한다.
- 다만 언어 표준 관점에서는 SystemVerilog 함수로 보는 편이 더 정확하므로, 순수 Verilog 환경에서는 수동 계산이나 별도 함수로 대체하는 경우도 있다.

### 형식만 먼저 보기

```verilog
localparam ADDR_W = $clog2(DEPTH);
reg [ADDR_W-1:0] addr;
```

### 예제

```verilog
module mem_if_example;
    parameter DEPTH = 256;
    localparam ADDR_W = $clog2(DEPTH);

    reg [ADDR_W-1:0] addr;

    initial begin
        $display("DEPTH=%0d ADDR_W=%0d", DEPTH, ADDR_W);
    end
endmodule
```

### 해석

- `DEPTH=256`이면 `ADDR_W=8`이다.
- `DEPTH=10`이면 `ADDR_W=4`가 된다.
- 고정 숫자를 손으로 세는 대신 폭을 자동 계산하므로, 파라미터화된 설계에서 특히 편하다.

### 7.4 실수 수학 함수는 어떻게 보면 되는가

`$sqrt`, `$pow`, `$exp`, `$ln`, `$log10`, `$sin`, `$cos`, `$atan2`는 대부분 "실수 계산을 시뮬레이션에서 보조하는 함수"로 이해하면 된다.

- 이 함수들은 보통 `real`과 함께 읽는 편이 자연스럽다.
- 테스트벤치에서 파형 비슷한 입력을 만들거나, 계산 결과를 빠르게 확인하거나, 계수 값을 점검할 때 유용하다.
- 반면 합성 RTL 안에서 직접 쓰는 경우는 드물다. 실제 하드웨어로 만들려면 고정소수점, LUT, 근사식, 미리 계산한 상수 등으로 다시 풀어야 하는 경우가 많다.
- 그래서 수업에서 "exp로 벡터를 만들 수는 있지만, 대부분 직접 작성하거나 미리 계산한 값을 넣는다"는 식으로 들렸다면 그 방향이 맞다.
- 즉, 실수 수학 함수는 "시뮬레이션과 검증의 편의 도구"로 보고, 합성 가능 회로 문법과는 조금 거리를 두는 편이 안전하다.

### 형식만 먼저 보기

```verilog
real x, y;
integer q;

y = $sqrt(x);
y = $pow(x, 2.0);
y = $exp(x);
y = $log10(x);
q = $rtoi(y);
```

### 예제

```verilog
integer i;
real    x, y;
integer vec;

initial begin
    for (i = 0; i < 5; i = i + 1) begin
        x = i;
        y = 255.0 * $exp(-0.5 * x);
        vec = $rtoi(y);
        $display("i=%0d exp=%0f vec=%0d", i, y, vec);
    end
end
```

### 해석

- 위 예제는 `$exp`로 감쇠 형태의 값을 만든 뒤, `$rtoi`로 정수 벡터 비슷하게 바꿔 보는 흐름이다.
- 테스트벤치에서는 이렇게 입력 시퀀스를 만들 수 있지만, 실제 RTL 설계에서는 보통 더 단순하고 명시적인 입력을 쓰는 편이 많다.
- 그래서 초반 학습 단계에서는 "이런 함수가 있다"를 알고, 실제 설계에서는 합성 가능성과 가독성을 먼저 보는 습관이 중요하다.

## 8. 여러 태스크를 한 번에 쓰는 테스트벤치 예제

아래 예제는 콘솔 출력과 파일 로그를 같이 쓰는 흐름이다.

### 문법만 먼저 보기

```verilog
$display(...)
$write(...)
$strobe(...)
$monitor(...)

fd = $fopen(...)
$fdisplay(fd, ...)
$fwrite(fd, ...)
$fclose(fd)
```

### 예제

```verilog
module tb_log_demo;
    reg        clk;
    reg  [3:0] a, b;
    wire [4:0] sum;
    integer    fd;

    assign sum = a + b;

    initial clk = 1'b0;
    always #5 clk = ~clk;

    initial begin
        fd = $fopen("tb_log_demo.log", "w");
        if (fd == 0) begin
            $display("file open failed");
            $finish;
        end

        $monitor("MON t=%0t a=%0d b=%0d sum=%0d", $time, a, b, sum);
        $fdisplay(fd, "simulation start");

        a = 4'd3; b = 4'd5;
        #10;
        $display("DISP t=%0t a=%0d b=%0d sum=%0d", $time, a, b, sum);
        $write("WRITE partial line: ");
        $write("a=%0d b=%0d ", a, b);
        $write("sum=%0d\n", sum);
        $strobe("STROBE settled sum=%0d", sum);
        $fdisplay(fd, "CASE1 t=%0t a=%0d b=%0d sum=%0d", $time, a, b, sum);

        a = 4'd7; b = 4'd9;
        #10;
        $fwrite(fd, "CASE2 ");
        $fwrite(fd, "t=%0t ", $time);
        $fwrite(fd, "a=%0d b=%0d sum=%0d\n", a, b, sum);

        $monitoroff;
        #10;
        $fdisplay(fd, "simulation end");
        $fclose(fd);
        $finish;
    end
endmodule
```

### 해석

- `$display`: 콘솔에 한 줄 출력
- `$write`: 콘솔에 줄 이어서 출력
- `$strobe`: 현재 time slot 끝의 값을 출력
- `$monitor`: 값 변화가 있을 때 자동 출력
- `$fdisplay`: 파일에 한 줄 출력
- `$fwrite`: 파일에 줄 이어서 출력
- `$fopen`, `$fclose`: 파일 열기와 닫기
- `$fgets`, `$sscanf`: 텍스트 라인 입력과 파싱
- `$swrite`, `$sformat`: 문자열 버퍼 생성
- `$readmemh`, `$readmemb`, `$fread`: 메모리/블록 입력 계열

즉 이 한 예제와 표들을 같이 보면, "콘솔 출력", "텍스트 파일 로그", "파일 입력", "메모리 로드", "고급 타이밍 계열"의 큰 지형을 한 번에 잡을 수 있다.

## 9. 그 외 자주 같이 보는 것

이 장의 중심은 아니지만 같이 자주 나온다.

| 항목 | 역할 |
| --- | --- |
| `$time` | 현재 시뮬레이션 시간 |
| `$stop` | 시뮬레이션 일시 중지 |
| `$finish` | 시뮬레이션 종료 |

## 자주 하는 실수

- 시스템 태스크를 합성 RTL 핵심 로직처럼 생각한다.
- `$display`와 `$write`의 자동 줄바꿈 차이를 잊는다.
- `$strobe`가 왜 필요한지 모르고 `$display`만 쓴다.
- `$monitor`를 켜 놓고 로그가 너무 많아져도 끄지 않는다.
- 파일을 열고 `$fclose`를 하지 않는다.
- `$fdisplay`와 `$fwrite`의 차이를 구분하지 않는다.
- 텍스트 입력에 `$fread`와 `$fscanf`를 혼동한다.
- `$sscanf`를 파일 입력과 같은 층위로 보지 못하고 문자열 파싱 도구라는 점을 놓친다.
- `$sformat`을 콘솔 출력 계열로 오해하고, 문자열 생성용이라는 점을 놓친다.
- `$rewind`, `$ftell`, `$fflush` 같은 파일 제어 계열을 아예 모른 채 불편하게 우회한다.
- `$random`만 돌리면 코너 케이스 검증도 충분하다고 착각한다.
- 자료형 변환 함수를 쓰면서 signed / unsigned 해석 차이를 놓친다.
- `$sdf_annotate`를 일반 콘솔 출력 계열처럼 생각한다.
