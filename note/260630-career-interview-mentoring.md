# 26-06-30 - 선배와의 직무/면접 상담

## 수업 흐름

0629 발표 피드백 이후, 현업에 들어간 선배들의 경험을 기준으로 반도체 설계 직무, 프로젝트 준비, 자기소개서와 면접 대응 방향을 정리했다. 핵심은 Verilog만 잘하는 것이 아니라 `C`, `Python`, `SystemVerilog`, Linux 환경, EDA tool 사용 흐름을 함께 가져가야 한다는 점이다.

현업에서는 설계 자체보다 검증, 자동화, 디버깅, 문서화, 발표와 설명 능력이 계속 요구된다. 수업 프로젝트도 단순히 동작 결과를 만드는 데서 끝내지 않고, 왜 그렇게 설계했는지, 어떤 문제가 있었는지, 어떤 근거로 해결했는지를 면접에서 설명할 수 있게 정리해야 한다.

## 현업 SoC 설계와 검증 흐름

SoC 팀은 CPU나 여러 IP를 단순히 따로 만드는 것이 아니라, IP와 bus, interconnect, memory map, peripheral을 엮어 하나의 chip 수준 시스템으로 구성한다. 설계가 커질수록 손으로 모든 연결을 작성하기 어렵기 때문에 Python 자동화와 generator 성격의 script가 많이 쓰인다.

| 단계 | 현업에서 보는 핵심 |
| :--- | :--- |
| IP/Block 설계 | Verilog/SystemVerilog 이해와 interface 연결 |
| 통합 | bus, interconnect, wrapper, register map 구성 |
| 기능 검증 | testbench, UVM, C golden model, simulation |
| 합성 | RTL을 gate-level netlist로 변환 |
| STA | timing constraint 기준으로 setup/hold 확인 |
| LEC | RTL과 합성 결과의 논리 등가성 확인 |
| post-simulation | timing 정보가 들어간 상태에서 재검증 |
| tape-out 이후 | chip 제작 중에도 문제 가능성 검토와 ECO 대응 |

수업에서 지금까지 다룬 Vivado, FPGA, AXI, UART, SPI/IIC, MicroBlaze 같은 흐름은 현업 SoC 설명의 기반이 된다. 다만 현업에서는 tool과 규모가 더 커지고, Linux server, VCS, Verdi, Synopsys/Cadence 계열 tool, git 기반 협업 흐름을 함께 다루게 된다.

## 언어별 준비 방향

| 언어/도구 | 준비 방향 |
| :--- | :--- |
| `Verilog` | RTL 구조를 읽고 block diagram을 코드로 구현할 수 있어야 함 |
| `SystemVerilog` | 검증과 testbench 흐름 이해 |
| `C` | golden model, test model, driver, embedded/SOC software 이해 |
| `Python` | register/interface 자동화, 반복 simulation, log/data 처리 |
| Linux shell | server 환경에서 command 중심으로 작업 |
| `git` | `add`, `commit`, `push`, conflict 처리와 팀 협업 |

선배들이 강조한 것은 `Verilog`는 기본 언어이고, `C`는 공용어에 가깝다는 점이다. UVM에서도 DPI-C처럼 C와 연결되는 흐름이 있고, C golden model과 RTL 결과를 비교하는 방식도 자주 쓰인다. driver code도 IP package와 함께 제공되는 경우가 많으므로, C를 완전히 피하면 현업 코드를 읽고 사용하는 데 막힌다.

Python은 단순 문법 문제가 아니라 자동화 능력으로 연결된다. 큰 SoC에서는 port, wire, register, testcase, library 선택, 반복 실행 결과 정리를 손으로 처리하기 어렵다. 그래서 Python으로 반복 작업을 줄이고, 필요한 데이터를 찾아 검증하는 능력이 중요하다.

## AI 활용 기준

AI는 적극적으로 사용할 수 있지만, 학습 단계에서 AI가 완성한 Verilog를 그대로 받아 쓰면 구조를 설명하지 못하게 된다. 면접에서는 내가 왜 이렇게 설계했는지, 왜 이런 분주값을 썼는지, 문제가 생겼을 때 어떻게 고쳤는지를 묻기 때문에, 코드의 완성도보다 설계 과정을 이해하는 것이 중요하다.

현업에서는 보안 때문에 회사 code를 외부 AI에 올릴 수 없고, 인터넷이 안 되는 server 환경에서 작업하는 경우도 많다. 따라서 AI를 쓰더라도 프로젝트 이름과 block 이름을 추상화하거나, Python 자동화 방향을 잡는 식으로 제한적으로 활용하게 된다.

| 상황 | 권장 태도 |
| :--- | :--- |
| 학습 중 RTL 작성 | AI 없이 block diagram부터 직접 구현 |
| 막힌 개념 확인 | 개념, 구조, debug 관점 질문 |
| Python 자동화 | 반복 작업 구조화에 적극 활용 |
| 회사 code | 외부 AI 업로드 금지 기준 준수 |
| 면접 준비 | AI 답변 암기보다 내 프로젝트 설명 능력 확보 |

## 프로젝트 정리와 발표 방향

프로젝트는 기술을 많이 나열하는 것보다 목표, 역할, 문제, 해결 근거, 결과를 설명할 수 있게 정리해야 한다. 같은 수업을 들은 학생들이 비슷한 산출물을 만들기 때문에, 발표에서는 내가 어떤 판단을 했고 어떤 차이를 만들었는지가 드러나야 한다.

| 정리 항목 | 면접에서 필요한 이유 |
| :--- | :--- |
| 프로젝트 목표 | 왜 이 주제를 했는지 설명 |
| 달성 수준 | 처음 목표 대비 무엇을 완성했는지 설명 |
| 본인 역할 | 팀 프로젝트에서 직접 한 부분 구분 |
| 사용 기술 | Verilog, C, Python, FPGA, tool 흐름 연결 |
| 문제 상황 | 실제로 부딪힌 bug와 원인 설명 |
| 해결 과정 | debug 방법과 근거 설명 |
| 검증 방법 | simulation, board demo, log, waveform 근거 |
| 기간과 소속 | 자기소개서와 이력서 작성 기준 |

기술 발표에서는 `무엇을 만들었다`보다 `왜 그렇게 만들었고, 어디서 문제가 났고, 어떻게 확인했는지`가 더 중요하다. 발표 자료에는 system architecture, block diagram, register map, 검증 시나리오, board evidence를 구분해서 넣어야 한다.

## 면접 질문 대비

면접 질문은 자기소개서와 프로젝트를 기반으로 깊게 들어온다. 특히 작은 회사일수록 프로젝트 세부 구현을 더 자세히 물어볼 수 있다. 모르는 질문이 나오면 무리해서 꾸미기보다, 아는 범위와 모르는 범위를 구분해서 답하는 태도가 필요하다.

예상 질문은 다음처럼 정리할 수 있다.

| 질문 유형 | 준비할 답변 |
| :--- | :--- |
| 프로젝트를 왜 했는가 | 주제 선정 이유와 해결하려던 문제 |
| 어떤 module을 설계했는가 | block별 역할과 interface |
| 검증은 어떻게 했는가 | testbench, testcase, waveform, board 확인 |
| `SystemVerilog`를 왜 썼는가 | 검증 구조와 class/task/interface 활용 이유 |
| UART baudrate는 왜 그렇게 잡았는가 | clock, 분주, 안정성 기준 |
| 문제가 생겼을 때 어떻게 해결했는가 | 원인 추적과 수정 근거 |
| 외부 환경이 바뀌면 어떻게 대응하는가 | threshold, register 설정, 재합성 없이 바꾸는 방법 |
| 내가 맡지 않은 부분은 아는가 | 팀 전체 구조와 내 역할 경계 |

면접관이 프로젝트를 몰라서 묻는 경우도 있다. 그래서 발표하듯이 전체 구조를 설명하고, 질문이 들어오면 하위 module과 signal 흐름으로 내려가는 연습이 필요하다.

## 자기소개서와 회사 조사

지원 동기는 가장 쓰기 어렵지만 가장 차별화할 수 있는 항목이다. 큰 회사는 공개 자료가 많지만, 작은 회사는 일반 검색으로 정보가 잘 나오지 않을 수 있다. 이때 뉴스, 사업보고서, 재무제표, 특허, 회사 홈페이지를 함께 확인하면 회사가 어떤 기술과 시장을 중요하게 보는지 잡을 수 있다.

| 자료 | 활용 방향 |
| :--- | :--- |
| 회사 홈페이지 | 제품군, 고객, 사업 영역 |
| 뉴스/기사 | 최근 수주, 투자, 기술 방향 |
| 특허 | 회사가 강조하는 기술 역량 |
| 사업보고서/재무제표 | 매출 구조와 사업 리스크 |
| 채용공고 | 요구 기술과 직무 키워드 |

자기소개서에는 모든 역량을 완벽히 갖췄다고 쓰기보다, 내가 가진 강점이 직무 요구사항과 어떻게 연결되는지를 보여준다. 부족한 점만 보고 지원을 접기보다, 내가 할 수 있는 영역을 중심으로 지원 범위를 넓게 잡는 것이 좋다.

## 취업 준비 태도

취업 준비는 수업과 프로젝트가 끝난 뒤 시작하는 것이 아니라 병행해야 한다. 완벽히 준비된 다음 지원하겠다는 생각은 시간을 늦춘다. 프로젝트를 진행할 때마다 바로 정리하고, 발표가 끝난 날에는 질문, 문제, 해결 과정, 맡은 역할을 기록해 두는 것이 좋다.

| 항목 | 실천 기준 |
| :--- | :--- |
| 프로젝트 기록 | 당일 정리, 질문과 답변 보존 |
| 포트폴리오 | Notion, PPT, 문서, QR link 등 접근 가능한 형태 |
| 발표 연습 | 기술 설명과 의사결정 근거 중심 |
| 교수 상담 | 직무 방향과 추천 기업 확인 |
| 지원 범위 | 설계, 검증, embedded/SOC software, P&R/PD까지 넓게 검토 |
| 멘탈 관리 | 부족한 점보다 장점과 지속성에 집중 |

선배 조언의 핵심은 `버티기`와 `병행`이다. 수업 시간만 채우고 끝내기보다 저녁과 주말 시간을 활용해 직접 다시 구현해 보고, 교수님 코드나 예제를 지운 뒤 스스로 다시 짤 수 있을 정도까지 반복하는 방식이 효과적이다.

## 직무 선택 메모

반도체 설계 직무는 매력적이지만 초반에는 업무 강도가 높을 수 있다. 신입 입사 후에는 OJT, 발표, 과제, tool 사용 훈련이 이어지고, 회사에 따라 1개월에서 6개월 이상 적응 기간을 둔다.

설계만 고집하기보다 검증, embedded/SOC software, P&R/PD도 같이 열어두면 선택지가 넓어진다. Verilog를 읽을 줄 알고 FPGA와 board 실습 경험이 있으며 C/Python을 함께 다룰 수 있다면, 여러 직무에서 반도체 시스템 이해도를 어필할 수 있다.
