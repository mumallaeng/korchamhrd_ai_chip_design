# Verilog HDL 복습노트 04 - SystemVerilog 검증 접근법

## 한 줄 요약

SystemVerilog 검증은 `transaction / generator / driver / monitor / scoreboard / environment`를 먼저 외우는 것이 아니라, **RTL이 약속한 동작을 먼저 적고 그에 맞는 구조를 고르는 것**이 핵심이다.

## 검증을 시작할 때 먼저 할 일

| 순서 | 질문 | 의미 |
| --- | --- | --- |
| 1 | DUT가 무슨 동작을 약속하는가 | reset, 저장, 출력, 상태 변화를 문장으로 적는다 |
| 2 | 무엇을 비교해야 하는가 | 단순 계산인지, 이전 상태를 기억해야 하는지 정한다 |
| 3 | 언제 넣고 언제 읽어야 하는가 | 입력 인가 시점과 출력 샘플 시점을 정한다 |
| 4 | 구조를 얼마나 나눌 것인가 | 작은 DUT면 단순 TB, 커지면 class 기반 구조로 확장한다 |

## 검증의 기본 루프

| 단계 | 내용 |
| --- | --- |
| 자극 생성 | 어떤 입력을 줄지 정한다 |
| DUT 구동 | 실제 핀에 값을 넣는다 |
| 출력 관찰 | 올바른 시점에 DUT 출력을 읽는다 |
| 기대값 계산 | 이 입력에서 어떤 결과가 나와야 하는지 계산한다 |
| 비교 | 기대값과 실제값을 비교한다 |
| 종료/집계 | pass/fail, 반복 횟수, 종료 조건을 관리한다 |

즉 검증의 본질은 `6개 블록`이 아니라 이 루프다.

## `register_8`에서 먼저 뽑아야 하는 검증 항목

`register_8.sv`는 아래 동작을 약속한다.

```verilog
always_ff @(posedge clk, posedge rst) begin
    if (rst) begin
        q <= 0;
    end else begin
        q <= d;
    end
end
```

여기서 바로 나오는 검증 포인트는 다음과 같다.

| 항목 | 확인할 것 |
| --- | --- |
| reset 동작 | `rst=1`이면 `q=0`이 되는가 |
| 저장 동작 | `posedge clk` 뒤에 `q`가 `d`를 따라가는가 |
| 유지 동작 | 클럭 사이에는 `q`가 함부로 바뀌지 않는가 |
| 반복 동작 | 여러 입력값에 대해서도 계속 같은 규칙이 유지되는가 |

## 이 DUT에서 필요한 기대값 모델

| DUT 종류 | 기대값 모델 |
| --- | --- |
| 조합회로 | 입력만으로 바로 계산 가능 |
| register | `expected_q` 한 변수면 충분 |
| RAM | 주소별 저장 상태를 위한 `mem[]` 필요 |
| FIFO | 순서를 기억하는 `queue` 필요 |

즉 `register_8`은 상태가 단순하므로 scoreboard도 아주 단순하게 만들 수 있다.

## 구조는 DUT에서 거꾸로 나온다

| RTL을 보고 먼저 정할 것 | 그다음 나오는 구조 |
| --- | --- |
| 샘플링 시점이 중요하다 | monitor를 따로 두는 것이 유리 |
| 기대값 계산이 복잡하다 | scoreboard를 독립시키는 것이 유리 |
| 랜덤 테스트를 많이 돌리고 싶다 | transaction, generator가 유리 |
| DUT 핀 구동을 분리하고 싶다 | driver가 유리 |
| 종료 조건과 반복 관리가 필요하다 | environment가 유리 |

즉 `6개를 무조건 다 쓴다`가 아니라, **필요할 때 블록을 꺼내 쓴다**가 맞다.

## 같은 DUT도 여러 방식으로 검증할 수 있다

| 방식 | 구조 | 언제 적합한가 |
| --- | --- | --- |
| 최소형 TB | `initial`에서 직접 `rst`, `d`, `q` 비교 | 아주 작은 DUT, 개념 확인용 |
| task 기반 TB | `do_reset()`, `drive_and_check()` 분리 | 절차를 조금 정리하고 싶을 때 |
| class 기반 TB | `transaction`, `mailbox`, `event`, scoreboard 분리 | 반복 테스트, 재사용, 구조화가 중요할 때 |

## `interface`와 `class`의 역할 차이

| 항목 | `interface` | `class` |
| --- | --- | --- |
| 본질 | DUT와 TB가 공유하는 실제 신호 묶음 | 검증 로직과 데이터를 담는 객체 |
| 대표 내용 | `clk`, `rst`, `d`, `q` | transaction, queue, counter, randomize |
| 시간축과 연결 | 있음 | 직접 없음 |
| DUT 포트 연결 | 가능 | 직접 불가 |
| 주 역할 | 핀/배선/타이밍 공유 | testcase 생성, 비교, 흐름 제어 |

한 줄로 정리하면:

- `interface` = 핀과 타이밍의 세계
- `class` = 데이터와 검증 절차의 세계

## 왜 class가 interface를 대체할 수 없는가

| 상황 | 설명 |
| --- | --- |
| class 안에 `d` 변수를 둠 | 그냥 변수일 뿐, DUT 입력 핀이 아님 |
| DUT에 실제 값을 넣고 싶을 때 | 결국 `virtual interface`를 통해 `reg_vif.d = tr.d;`처럼 복사해야 한다 |
| DUT 출력을 읽고 싶을 때 | 결국 `reg_vif.q`처럼 interface 쪽 실제 신호를 읽어야 한다 |

즉 class는 값을 **기억하고 판단**할 수 있지만, DUT와 직접 연결된 신호 공간은 interface 쪽이다.

## `interface`, `virtual interface`, `transaction(item)` 관계

| 항목 | 의미 |
| --- | --- |
| `interface` | testbench top에 실제로 존재하는 신호 묶음 |
| `virtual interface` | class 안에서 그 interface를 가리키는 참조(handle) |
| `transaction` | 검증에 쓰는 데이터 묶음이라는 역할 이름 |
| `item` | 그 transaction을 구현한 실제 class 이름 |

한 줄로 정리하면:

- `interface` = 실제 신호와 타이밍의 세계
- `virtual interface` = class가 그 신호 세계에 접근하기 위한 손잡이
- `transaction(item)` = 검증 데이터를 담는 객체

즉 `transaction`은 DUT 안으로 들어가는 것이 아니라, **TB 클래스 쪽에서만 존재하는 데이터 객체**다.

## 실제 연결 구조는 이렇게 본다

```text
generator -> item(transaction) -> mailbox -> driver -> virtual interface -> interface -> DUT
DUT -> interface -> virtual interface -> monitor -> observed item(transaction) -> mailbox -> scoreboard
```

여기서 중요한 점:

- DUT 안팎을 오가는 것은 항상 **실제 신호**
- mailbox로 오가는 것은 항상 **item(transaction)**
- `driver`, `monitor`는 class라서 DUT 핀에 직접 붙는 것이 아니라 `virtual interface`를 통해 interface 신호를 읽고 쓴다

## 그래서 `item`은 어디에 있는가

| 질문 | 답 |
| --- | --- |
| `item`이 generator 안에 있는가 | generator가 잠깐 만들어서 들고 있는 것일 뿐, generator 자체가 item은 아님 |
| `item`이 driver 안에 있는가 | driver가 mailbox에서 받아 쓰는 것일 뿐, driver 자체가 item은 아님 |
| `item`이 DUT 안에 있는가 | 아님. DUT는 item을 모르고 신호만 봄 |

즉 정확한 표현은:

- generator가 `item(transaction)`을 만든다
- driver가 그 item을 받아 interface 신호로 바꿔 DUT에 넣는다
- monitor가 DUT 신호를 읽어 observed item으로 다시 묶는다

## `transaction`과 `item`은 같은 말인가

거의 같은 말처럼 써도 된다. 다만 뉘앙스가 다르다.

- `transaction`
  - 역할 이름
  - 검증용 데이터 묶음이라는 개념
- `item`
  - 구현 이름
  - 예: `fifo_item`, `uart_rx_item`, `uart_stream_item`

즉 문서에서:

- `fifo_item`
- `uart_rx_item`
- `uart_stream_item`

이런 것들은 전부 `transaction class`라고 봐도 된다.

## `tb_register_8.sv`를 이 관점으로 읽기

| 블록 | 이 파일에서 하는 일 |
| --- | --- |
| `transaction` | 한 번의 입력 `d`와 관측값 `q`를 묶는다 |
| `generator` | 여러 `d` 값을 만들어 mailbox로 보낸다 |
| `driver` | `reg_vif.d`에 실제 값을 넣는다 |
| `monitor` | `posedge clk` 뒤 `#1`에서 `q`를 읽는다 |
| `scoreboard` | `d == q`인지 비교한다 |
| `environment` | reset, 반복 횟수, 종료 조건을 관리한다 |

여기서 중요한 것은 이름이 아니라:

- 왜 `driver`와 `monitor`를 분리했는가
- 왜 `monitor`가 `posedge` 뒤 `#1`에서 읽는가
- 왜 `scoreboard`가 단순 비교만 해도 되는가

를 이해하는 것이다.

## 실제로는 이렇게 생각하면 된다

| 단계 | `register_8`에서의 질문 |
| --- | --- |
| 1 | reset이 있나? 있다면 우선 reset 케이스를 검증해야 한다 |
| 2 | 출력은 언제 바뀌나? `posedge clk` 뒤에 바뀐다 |
| 3 | 기대값은 복잡한가? 아니다. `expected_q = d`면 된다 |
| 4 | 랜덤 반복이 필요한가? 필요하면 generator를 둔다 |
| 5 | 샘플 시점이 민감한가? 그렇다. monitor를 따로 두는 편이 좋다 |

즉 **`RTL 계약 -> 검증 항목 -> 샘플링 시점 -> 필요한 블록 선택`** 순서로 가면 된다.
