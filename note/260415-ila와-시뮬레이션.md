# 26-04-15 - ILA와 시뮬레이션의 역할 구분

## 한눈에 보기

| 항목 | 내용 |
| --- | --- |
| 목표 | ILA를 `시뮬레이터 대체재`가 아니라 `보드 내부 파형 캡처 도구`로 이해한다 |

## ILA를 이렇게 보면 된다

| 항목 | 정리 |
| --- | --- |
| 역할 | FPGA 내부 신호를 보드에서 직접 캡처 |
| 입력 | `clk`와 여러 `probe` |
| 결과 | Hardware Manager에서 내부 파형 확인 |
| 한계 | stimulus 생성, exhaustive test, 빠른 반복 검증은 시뮬레이션이 더 강하다 |

## 현재 저장소 기준 해석

| 항목 | 정리 |
| --- | --- |
| `sr04.v` | 버튼, 거리 측정, FND 표시 같은 실제 기능 경로가 있다 |
| `ila_0_stub.v` | ILA IP의 입출력 형태만 보여 준다 |
| 주의 | 현재 top source에는 ILA 인스턴스가 직접 보이지 않으므로 generated IP만 있다고 끝난 것이 아니다 |

## 시뮬레이션과 ILA 차이

| 구분 | 시뮬레이션 | ILA |
| --- | --- | --- |
| 목적 | 설계 논리 검증 | 보드 내부 실시간 디버깅 |
| 입력 자극 | 자유롭게 만든다 | 실제 보드 입력에 의존 |
| 반복 속도 | 빠르다 | 느릴 수 있다 |
| 내부 신호 | 원하는 만큼 관찰 가능 | probe로 잡은 것만 본다 |

## 주의점

| 실수 | 정리 |
| --- | --- |
| ILA만 쓰면 된다고 본다 | 기본 검증은 여전히 testbench다 |
| probe를 너무 늦게 정한다 | 미리 어떤 신호를 볼지 정해야 한다 |
| generated IP를 추가한 것과 top 연결을 같은 것으로 본다 | 둘은 별개다 |
