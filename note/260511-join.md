# JOIN

## fork-join 요약

| 구문 | 의미 | 다음 문장 진행 시점 |
| --- | --- | --- |
| `fork ... join` | 병렬 스레드를 모두 실행 | 모든 스레드 종료 후 |
| `fork ... join_any` | 병렬 스레드를 모두 실행 | 하나라도 종료되면 진행 |
| `fork ... join_none` | 병렬 스레드를 백그라운드로 실행 | 즉시 진행 |
| `disable fork` | 현재 fork 블록에서 생성된 자식 스레드 종료 | 호출 즉시 |

## task 정리

| 항목 | 정리 |
| --- | --- |
| task 재정의 | 같은 module 안에서 같은 이름 task를 다시 정의하지 않음 |
| 권장 방식 | 공통 task를 만들고 인자로 이름, 반복 횟수, 주기 등을 전달 |
| 주의점 | `fork` 안의 실행 순서는 task 선언 순서가 아니라 `#delay`, `join`, `join_any`, `join_none`이 결정 |

# SRAM

## 개념 정리

| 항목 | 내용 |
| --- | --- |
| RAM | 어느 주소든 같은 방식으로 바로 접근할 수 있는 메모리 |
| SRAM | 전원이 있는 동안 저장값을 refresh 없이 유지하는 RAM |
| `ram_ip.sv` | SRAM 칩 자체라기보다 RAM처럼 동작하는 저장소 모델로 이해 |
| 구조 | single-port RAM, sync write, async read |
| 검증 포인트 | 메모리는 과거 write 이력에 따라 read 값이 결정되므로 shadow memory 기반 비교가 필요 |

## `ram_ip.sv` 핵심 코드

```systemverilog
logic [7:0] ram[0:255];

always_ff @(posedge clk) begin
    if (we) ram[addr] <= wdata;
end

always_comb rdata = ram[addr];
```

## 테스트벤치 흐름

| 블록 | 역할 |
| --- | --- |
| `GEN` | `addr`, `wdata`, `we`를 생성 |
| `DRV` | mailbox에서 transaction을 받아 DUT에 인가 |
| `MON` | 인터페이스 신호를 관찰해 실제 동작값 캡처 |
| `SCB` | shadow memory `mem[addr]` 기준으로 기대값과 실제값 비교 |

## SRAM 검증 시나리오

| 라벨 | 설명 | 자극 생성 | 검증 포인트 |
| --- | --- | --- | --- |
| `RAM_INIT_IDLE` | 시작 직후 기본 버스 상태 확인 | `driver.preset()` 후 초기 몇 클럭 관찰 | `addr=0`, `wdata=0`, `we=0` 상태와 비교 흐름 정상 여부 |
| `RAM_WRITE_PATH` | write 경로 확인 | `we=1` 중심 directed 또는 randomized stimulus | DUT 입력 전달, shadow memory 갱신, 주소별 저장 여부 |
| `RAM_READBACK` | 저장된 값 readback 확인 | 먼저 write로 메모리 채운 뒤 `we=0` read-only 구간 수행 | `rdata == mem[addr]` 비교 |
| `RAM_RANDOM_MIX` | write/read 혼합 동작 확인 | `addr`, `wdata`, `we` 랜덤화 반복 | write 후 read, 연속 write, read-only 접근까지 전체 흐름 점검 |

# FIFO

## 구현 방향

| 파일 | 역할 |
| --- | --- |
| `fifo.sv` | DUT. 배열, pointer, 상태 신호로 FIFO 동작 구현 |
| `tb_fifo.sv` | 검증용 testbench |
| reference model | SystemVerilog queue를 golden model로 사용 |

## FIFO 핵심 구조

| 블록 | 역할 |
| --- | --- |
| `register_file` | 실제 데이터 저장 |
| `control_unit` | `wptr`, `rptr`, `full`, `empty` 제어 |
| `push` | `!full`일 때 write 수행 |
| `pop` | `!empty`일 때 read 수행 |

## queue 기반 reference model 예시

```systemverilog
logic [7:0] ref_q[$];

if (push && !full) begin
    ref_q.push_back(push_data);
end

if (pop && !empty) begin
    exp_data = ref_q.pop_front();
end
```

## queue 메서드

| 메서드 | 설명 |
| --- | --- |
| `push_back()` | queue 맨 뒤에 추가 |
| `push_front()` | queue 맨 앞에 추가 |
| `pop_front()` | queue 맨 앞에서 꺼내고 반환 |
| `pop_back()` | queue 맨 뒤에서 꺼내고 반환 |
| `size()` | 현재 queue 크기 반환 |
| `insert()` | 특정 위치에 삽입 |
| `delete()` | 특정 위치 요소 삭제 |

| FIFO_2 | Write-only sequence. | Constrain writing only and no read during simulation. | Cross coverage between write enable, read enable, and each output control signal. | A checker in the scoreboard checks the internal signals and counter. | FIFO_3 | Read-only sequence. | Constrain reading only and no write during simulation. | Cross coverage between write enable, read enable, and each output control signal. | A checker in the scoreboard checks the internal signals and counter. | FIFO_4 | Verify the functionality of the FIFO with constrained randomization. | Randomization in a repeat of 9999 iterations with constraints: reset off 90% of the time, write enable high 70% of the time, and read enable active 30% of the time. | Cross coverage between write enable, read enable, and each output control signal. | A checker in the scoreboard checks the internal signals and counter.
