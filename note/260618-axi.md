# 26-06-18 - AXI 시작

## 수업 흐름

오전에는 발표가 마무리되었고, 오후부터 AXI 수업이 시작되었다. 0618 수업의 중심은 AXI RTL 구현이 아니라 AMBA AXI4 문서 기준으로 AXI가 어떤 interface protocol인지, channel이 어떻게 나뉘는지, `VALID/READY` handshake가 어떤 의미인지 잡는 것이었다.

AXI는 SoC 내부 IP block 사이를 연결하기 위한 interface protocol이다. AXI 자체가 interconnect 전체를 의미하는 것은 아니며, master와 slave가 어떤 signal과 timing 규칙으로 정보를 주고받는지 정의하는 규격으로 봐야 한다. 여러 slave 중 어디로 요청을 보낼지는 address map 또는 transaction ID를 보고 interconnect/switch가 판단한다.

후반에는 AXI master를 먼저 잡고, 이후 slave를 붙여 master/slave simulation을 보는 흐름으로 설명되었다. 그 다음에는 Xilinx에서 제공하는 AXI 관련 예제나 IP 구조를 참고해 slave/register interface를 구현하고, MicroBlaze 같은 processor와 연결하는 방향으로 확장할 수 있다.

[Introduction to AMBA AXI4](https://developer.arm.com/documentation/ihi0022/l/)는 AXI 개념을 확인할 때 기준 문서로 두었다.

## AXI 핵심 개념

AXI(Advanced eXtensible Interface)는 IP block 사이의 interface를 정의하는 AMBA protocol이다. master와 slave interface 사이에서 어떤 signal을 쓰고 어떤 timing으로 정보를 주고받는지를 정의하며, 여러 master와 여러 slave를 연결하려면 별도의 interconnect fabric이 필요하다.

| 용어 | 의미 |
| :--- | :--- |
| AXI | IP block 사이 interface 규격 |
| Master | address/control/data 요청을 시작하는 쪽 |
| Slave | master 요청을 받아 read/write 응답을 내는 쪽 |
| Interconnect | 여러 master/slave interface 연결 fabric, address map 또는 ID 기준 routing |
| Transfer | 한 channel에서 `VALID && READY`가 성립한 1회 정보 이동 |
| Transaction | address, data, response까지 묶은 read/write 작업 단위 |

AXI에서 중요한 점은 channel이 독립되어 있다는 것이다. read와 write가 같은 신호 묶음으로 섞이지 않고, address, data, response가 channel 단위로 나뉜다.

| Channel | 방향 | 역할 |
| :--- | :--- | :--- |
| `AW` Write Address | Master -> Slave | write 주소와 control 정보 |
| `W` Write Data | Master -> Slave | write data와 byte strobe |
| `B` Write Response | Slave -> Master | write 완료 응답 |
| `AR` Read Address | Master -> Slave | read 주소와 control 정보 |
| `R` Read Data | Slave -> Master | read data와 read 응답 |

`Master/Slave`는 transaction을 누가 시작하고 누가 응답하는지에 대한 역할이고, `Source/Destination`은 각 channel에서 정보를 누가 내고 누가 받는지에 대한 역할이다. write address/data channel에서는 master가 source이고 slave가 destination이지만, write response와 read data channel에서는 slave가 source이고 master가 destination이 된다.

## VALID/READY handshake

AXI의 모든 channel은 `VALID`와 `READY` handshake를 사용한다. source는 보낼 정보가 유효할 때 `VALID`를 올리고, destination은 받을 준비가 되었을 때 `READY`를 올린다. clock rising edge에서 `VALID`와 `READY`가 동시에 high이면 해당 channel의 transfer가 완료된다.

| 신호 | 내는 쪽 | 의미 |
| :--- | :--- | :--- |
| `VALID` | Source | data/control 정보 유효 |
| `READY` | Destination | 수신 가능 상태 |
| `VALID && READY` | 양쪽 조건 동시 만족 | rising edge 기준 transfer 성립 |
| Sticky VALID | Source 유지 조건 | destination이 받을 때까지 유효 정보 유지 |

`READY`는 destination이 미리 high로 둘 수 있다. 이 경우 source가 `VALID`를 올리는 cycle에 바로 transfer가 끝날 수 있어 latency를 줄일 수 있다. 반대로 destination이 준비되지 않으면 `READY=0` 상태가 되고, source는 `VALID`와 payload를 유지해야 한다.

| 상황 | 동작 |
| :--- | :--- |
| `READY` 선출력 | source가 `VALID` 올리는 cycle에 바로 transfer 가능 |
| `VALID` 선출력 | destination이 `READY` 올릴 때까지 payload 유지 |
| 둘 다 low | 해당 channel transfer 없음 |
| `VALID && READY` | rising edge 기준 payload 수신 |

write response의 `BVALID`는 write address channel과 write data channel의 handshake가 끝난 뒤에 의미가 있다. read data의 `RVALID`도 read address channel handshake가 끝난 뒤에 나온다. 따라서 AXI waveform을 볼 때는 단일 signal 하나만 보지 않고, address/data/response channel의 handshake 순서를 같이 확인해야 한다.

## AXI write/read transaction 흐름

AXI에서 transfer는 한 channel의 정보 이동이고, transaction은 하나의 read 또는 write 작업 전체를 의미한다. 예를 들어 write transaction은 `AW`, `W`, `B` channel이 함께 만들어내는 작업이며, read transaction은 `AR`, `R` channel이 함께 만들어내는 작업이다.

| Transaction | Channel 흐름 | 확인할 handshake |
| :--- | :--- | :--- |
| Write | `AW` address, `W` data, `B` response | `AWVALID/AWREADY`, `WVALID/WREADY`, `BVALID/BREADY` |
| Read | `AR` address, `R` data/response | `ARVALID/ARREADY`, `RVALID/RREADY` |

write address와 write data는 동시에 진행될 수도 있고, 한쪽이 먼저 진행될 수도 있다. 중요한 기준은 각 channel에서 `VALID && READY`가 성립한 clock edge이며, slave는 필요한 address/data를 받은 뒤 write response를 돌려준다. read는 master가 `AR` channel로 address를 보낸 뒤 slave가 `R` channel에서 data와 response를 돌려주는 구조다.

## AXI4, AXI4-Lite, AXI4-Stream 구분

| 구분 | 특징 | 사용 관점 |
| :--- | :--- | :--- |
| AXI4 | burst 지원, 고성능 memory-mapped transfer | memory, DMA, 고대역폭 IP |
| AXI4-Lite | burst 없음, 단일 read/write transfer, `WLAST` 없음 | control/status register |
| AXI4-Stream | address 없음, data stream 중심 | 영상, packet, FIFO stream |

AXI4 full은 한 번 address를 보낸 뒤 여러 data beat를 burst로 보낼 수 있고, burst type에 따라 address가 증가하는 형태도 사용할 수 있다. AXI4-Lite는 한 address에 한 data를 주고받는 단순한 구조라 Custom IP의 register read/write에 자주 연결된다. address channel과 data channel, response channel은 유지하지만 burst가 없으므로 제어 register 접근을 설명하기 좋다. AXI4-Stream은 address 없이 data 흐름을 보내는 구조라 memory-mapped register 접근과는 관점이 다르다.

## AXI와 Custom IP 연결

AXI4-Lite는 processor 또는 interconnect가 Custom IP 내부 register를 read/write하는 interface로 자주 사용된다. Custom IP가 내부에 control register, status register, data register를 갖고 있으면, processor는 AXI4-Lite address map을 통해 해당 register에 접근한다.

| 관점 | AXI4-Lite에서 확인할 내용 |
| :--- | :--- |
| 선택 방식 | address map 기준 register 선택 |
| write 동작 | `AW` address, `W` data, `B` response |
| read 동작 | `AR` address, `R` data/response |
| 검증 포인트 | channel handshake, register write/read, response code |
| 확장 방향 | Xilinx AXI IP 예제, MicroBlaze 연결 |

따라서 AXI 수업의 다음 단계는 단순히 signal 이름을 외우는 것이 아니라, master가 어떤 address와 data를 보내고 slave가 어떤 register를 갱신하거나 read data를 돌려주는지 simulation에서 확인하는 것이다.

## 실습/과제 연결

0618 수업의 과제 방향은 AXI4-Lite slave module을 설계하고, master와 slave를 연결해 simulation으로 read/write 동작을 확인하는 것이다. 제출물에는 source code와 simulation capture가 필요하므로, 파형에서는 channel별 handshake와 register data 변화를 함께 보여야 한다.

| 확인 대상 | 파형에서 볼 신호 |
| :--- | :--- |
| Write address | `AWADDR`, `AWVALID`, `AWREADY` |
| Write data | `WDATA`, `WVALID`, `WREADY` |
| Write response | `BRESP`, `BVALID`, `BREADY` |
| Read address | `ARADDR`, `ARVALID`, `ARREADY` |
| Read data | `RDATA`, `RRESP`, `RVALID`, `RREADY` |
| 내부 register | write 후 register 값, read 시 `RDATA` 값 |

정확한 파형 시간과 기대값은 0619 AXI master/slave 소스를 실행한 뒤 확인해야 한다. 0618 노트에서는 아직 verified simulation 결과를 만들지 않고, capture target만 정리한다.
