# 26-06-02 - RV32I datapath, memory map, APB

RV32I CPU를 읽을 때는 instruction이 datapath를 움직이는 흐름에서 시작해 memory map과 bus 구조까지 이어서 봐야 한다.
CPU는 machine code를 fetch/decode한 뒤 control signal로 register file, immediate generator, ALU, memory path를 움직인다.
memory access 단계에서 만들어진 address와 read/write 요청은 memory map, decoder, chip select를 거쳐 memory 또는 peripheral로 전달된다.

## RV32I instruction과 datapath 흐름

instruction memory에는 machine code가 들어 있고, PC가 가리키는 address에서 instruction을 읽는다.
control unit은 `opcode`, `funct3`, `funct7`을 해석해 ALU 연산, register write, memory access, PC 갱신 같은 control signal을 만든다.
datapath는 이 control signal에 따라 register file, immediate generator, ALU, memory path, write-back mux를 움직인다.

| instruction 계열 | datapath에서 보는 핵심 |
| --- | --- |
| `R-type` | `rs1`, `rs2`를 ALU 입력으로 쓰고 결과를 `rd`에 write-back |
| `I-type` | ALU 두 번째 입력을 sign-extended immediate로 바꾸고 결과를 `rd`에 write-back |
| `load` | `rs1 + imm`로 data address를 만들고 memory read 결과를 `rd`에 write-back |
| `store` | `rs1 + imm`로 data address를 만들고 `rs2` 값을 write data로 전달 |
| branch/jump | ALU 비교 또는 target address 계산 결과로 다음 PC 후보를 선택 |

같은 ALU가 register-register 연산, immediate 연산, memory address 계산을 모두 처리한다.
따라서 `alu_src_sel` 같은 mux control은 ALU 두 번째 입력으로 `rs2`를 쓸지 immediate를 쓸지 결정하는 핵심 신호다.

## RV32I immediate 배치 기준

RV32I immediate bit는 사람이 읽기 쉬운 순서보다 hardware decode와 배선 단순화를 우선해 배치된다.
형식마다 immediate가 흩어져 보이지만 공통 기준은 일정하다.

| 기준 | 의미 |
| --- | --- |
| sign bit 고정 | sign extension을 단순하게 만들기 위해 immediate의 부호 bit는 공통적으로 `inst[31]`에서 온다 |
| register field 위치 고정 | `rd=inst[11:7]`, `rs1=inst[19:15]`, `rs2=inst[24:20]` 위치를 최대한 유지한다 |
| source bit 공유 | 여러 format이 `inst[30:25]`, `inst[11:8]`, `inst[19:12]` 같은 instruction bit 묶음을 재사용한다 |
| alignment 활용 | branch/jump target은 하위 1bit가 `0`인 2-byte 정렬 offset으로 보고 `imm[0]`을 instruction에 저장하지 않는다 |

즉 immediate generator는 bit를 순서대로 잘라내는 블록이 아니다.
명령어 형식별로 흩어진 bit를 재조립하고, sign extension과 하위 zero bit를 붙여 실제 ALU 입력이나 PC offset을 만드는 datapath 블록이다.

## Type별 immediate 구조

R-type은 immediate가 없고 `rs1`, `rs2`, `rd`를 모두 사용한다.
나머지 format은 어떤 register field가 필요 없는지에 따라 그 자리에 immediate를 넣는다.

| Type | immediate source | 배치 이유 |
| --- | --- | --- |
| `I-type` | `imm[11:0] = inst[31:20]` | `rs1`, `rd`를 유지하고 남는 상위 12bit를 immediate로 사용 |
| `S-type` | `imm[11:5] = inst[31:25]`, `imm[4:0] = inst[11:7]` | store는 `rs1` base와 `rs2` write data가 필요하고 `rd`가 없으므로 `rd` 자리까지 immediate로 사용 |
| `B-type` | `imm[12] = inst[31]`, `imm[10:5] = inst[30:25]`, `imm[4:1] = inst[11:8]`, `imm[11] = inst[7]`, `imm[0] = 0` | `rs1`, `rs2` 비교를 유지하면서 S-type과 비슷한 source 위치를 재사용 |
| `U-type` | `imm[31:12] = inst[31:12]`, `imm[11:0] = 0` | `LUI`, `AUIPC`처럼 상위 20bit를 크게 채우기 위해 `rd`를 제외한 상위 공간을 사용 |
| `J-type` | `imm[20] = inst[31]`, `imm[10:1] = inst[30:21]`, `imm[11] = inst[20]`, `imm[19:12] = inst[19:12]`, `imm[0] = 0` | `JAL`의 긴 PC-relative jump offset을 만들면서 sign bit와 정렬 규칙을 유지 |

`B-type`과 `J-type`에서 bit가 섞여 보이는 이유는 `offset << 1`을 별도 shifter처럼 처리하기보다, instruction bit를 필요한 immediate bit 위치에 가깝게 배치했기 때문이다.
인간이 보기에는 순서가 어색하지만 hardware 입장에서는 sign extension, register decode, immediate mux 배선이 단순해진다.

## single-cycle 경로와 단계 이름

single-cycle CPU에서는 instruction 하나가 아래 흐름을 한 clock 안에서 모두 지나간다.

| 단계 | 의미 | 대표 흐름 |
| --- | --- | --- |
| fetch | PC가 가리키는 instruction을 instruction memory에서 읽음 | `PC -> instruction memory -> instr_code` |
| decode | instruction bit를 해석하고 register/immediate를 준비 | `opcode/funct3/funct7 -> control`, `rs1/rs2 -> register file read` |
| execute | ALU 연산, address 계산, branch/jump 판단 | `rs1/rs2` 또는 `rs1/imm -> ALU` |
| memory | data memory 또는 peripheral address에 read/write 수행 | `address`, `write data`, `read/write enable` |
| write-back | ALU 결과, memory read data, PC 관련 값을 register file로 되돌림 | write-back mux -> `rd` |

이 전체 경로가 한 clock에 이어지면 clock period는 가장 긴 조합 경로에 묶인다.
특히 load/store는 register file read, ALU address 계산, memory 접근, write-back 경로가 길게 이어지므로 single-cycle의 한계를 잘 보여 준다.
multi-cycle 구조는 이 흐름을 state 단위로 나눠 중간 값을 register에 저장하고, pipeline 구조는 여러 instruction이 서로 다른 stage에 겹쳐 지나가게 만든다.

## memory access에서 memory map으로

CPU는 연결된 장치가 ROM인지 RAM인지 GPIO인지 직접 구분하지 않는다.
CPU 입장에서는 정해진 address에 읽거나 쓰는 동작만 수행한다.
그 address를 어느 장치에 연결할지는 memory map과 decoder가 결정한다.

| 개념 | 의미 |
| --- | --- |
| memory map | CPU address space 안에 memory/peripheral을 배치한 약속 |
| address decoder | address 상위 bit를 보고 선택 신호를 생성하는 회로 |
| chip select / peripheral select | 여러 장치 중 실제로 반응할 대상 선택 |
| register offset | 선택된 peripheral 내부에서 어떤 register를 볼지 정하는 하위 주소 |

memory map을 이해하면 "왜 이 주소에 쓰면 peripheral이 동작하는가"를 설명할 수 있다.
software는 load/store처럼 보지만 hardware에서는 address decode, bus transaction, selected register 접근으로 풀린다.

## schematic과 bus를 읽는 기본

회로도에서는 같은 이름의 net이 서로 연결된 것으로 읽는다.
address bus, data bus, read/write 신호는 여러 장치에 같이 뿌려지고, 선택 신호가 켜진 장치만 의미 있게 반응한다.

| 예 | 의미 |
| --- | --- |
| `A0`가 여러 칩에 반복됨 | 같은 address bit가 여러 장치에 같이 연결됨 |
| `D0`가 여러 칩에 반복됨 | 같은 data bit가 여러 장치에 같이 연결됨 |
| `RD`, `WR` | read/write 제어가 여러 장치에 같이 전달됨 |
| `CS`, `CE` | 실제로 어느 칩이 동작할지 고르는 선택 신호 |

`74LS138` 같은 3-to-8 decoder는 address bit 조합을 one-hot select로 바꾼다.
예시 칩처럼 low-active 출력이면 선택된 출력이 `0`이고 나머지가 `1`이다.
데이터시트에서 signal 이름 위에 bar가 있으면 active-low로 읽는다.

## address range와 MMIO

address 상위 bit로 ROM, RAM, I/O, reserved 영역을 나눌 수 있다.

| `A15:A13` | address range | 선택 대상 | 의미 |
| --- | --- | --- | --- |
| `000` | `0x0000` - `0x1FFF` | ROM | program 또는 고정 data 영역 |
| `001` | `0x2000` - `0x3FFF` | RAM | 실행 중 data 저장 영역 |
| `010` | `0x4000` - `0x5FFF` | 8255 I/O | peripheral I/O 영역 |
| 그 외 | `0x6000` - `0xFFFF` | reserved | 아직 장치를 배치하지 않은 영역 |

memory-mapped I/O는 peripheral을 memory address처럼 접근하는 구조다.
CPU는 peripheral 전용 명령을 따로 두기보다, 특정 address에 read/write를 수행한다.
그 address가 peripheral range에 들어가면 decoder가 해당 peripheral의 select를 켠다.

reserved는 숨은 memory가 아니라 현재 설계에서 사용하지 않거나 나중에 확장할 수 있도록 남긴 address 영역이다.
32-bit CPU가 4GB address space를 표현할 수 있어도 그 전체가 실제 RAM이라는 뜻은 아니다.

## SoC와 bus 구조

과거에는 ROM, RAM, I/O, UART, timer 같은 칩이 board 위에 따로 있었지만, 현재는 CPU, memory, peripheral이 하나의 chip 안으로 들어가는 SoC 구조가 중요해졌다.
SoC 회사는 CPU core IP를 가져다 쓰고, 제품에 필요한 peripheral을 설계해 address map과 bus에 배치할 수 있다.

| 설계자가 정해야 하는 것 | 의미 |
| --- | --- |
| 어떤 peripheral을 넣을지 | GPIO, timer, UART, I2C 등 제품 기능 결정 |
| 어느 address에 배치할지 | software가 접근할 register base address 결정 |
| 어떤 bus에 연결할지 | APB, AHB, AXI 같은 SoC bus 선택 |
| register map offset | base address 안에서 각 register가 어디 있는지 결정 |

AMBA 계열 bus 그림을 볼 때도 CPU 또는 master가 address를 내고, decoder/select가 slave/peripheral을 고른다는 큰 흐름은 같다.

## APB 구조와 transfer

APB는 control register 중심의 단순 peripheral 접근에 맞춘 bus다.
CPU 가까운 고속 영역은 보통 AHB/AXI 같은 system bus가 담당하고, GPIO/UART/timer 같은 느린 peripheral은 bridge를 거쳐 APB 쪽에 붙는다.

| bus 계층 | 역할 |
| --- | --- |
| AHB/AXI | CPU, memory, 고속 path에 가까운 system bus |
| APB bridge | system bus 요청을 APB peripheral 접근으로 변환 |
| APB | control register 중심의 단순 peripheral 접근 |

APB 전송은 `IDLE`, `SETUP`, `ACCESS`로 읽는다.

| phase | 주요 신호 | 의미 |
| --- | --- | --- |
| `IDLE` | `PSEL=0` | 전송 없음 |
| `SETUP` | `PSEL=1`, `PENABLE=0` | address, 방향, write data 준비 |
| `ACCESS` | `PSEL=1`, `PENABLE=1` | 실제 read/write 수행, `PREADY` 확인 |

`PREADY=0`이면 slave가 아직 준비되지 않은 것이므로 `ACCESS` 상태를 유지한다.
`PREADY=1`이면 이번 transfer가 완료되고, 다음 요청이 있으면 다음 setup으로 넘어간다.
APB는 한 transfer가 끝나야 다음 transfer로 넘어가는 비파이프라인 peripheral bus로 이해하면 된다.

## programmable control register

APB peripheral은 CPU가 program으로 제어할 수 있는 register 묶음으로 보면 된다.
software는 peripheral의 base address에 register offset을 더해서 특정 register를 읽거나 쓴다.

```text
peripheral register address = base address + offset
```

| 용어 | 의미 |
| --- | --- |
| base address | decoder가 해당 peripheral을 선택하게 만드는 기준 주소 |
| offset | 선택된 peripheral 내부에서 어떤 register를 볼지 정하는 상대 주소 |
| register map | base address 안에서 각 register offset을 정리한 표 |

address 상위 bit는 peripheral select에 쓰이고, 선택된 뒤의 하위 address bit는 peripheral 내부 register 선택에 쓰인다.
read에서는 여러 slave의 `PRDATA` 중 선택된 slave의 값만 master로 돌아가야 하고, ready도 선택된 slave의 `PREADY`를 기준으로 본다.

## 학습 포인트

아래 흐름을 한 줄로 읽을 수 있으면 RV32I datapath와 SoC bus가 연결된다.

```text
instruction format
-> control signal
-> immediate generator
-> datapath effective address
-> memory map / base address
-> register offset
-> bus transaction
-> address decoder
-> chip select / peripheral select
-> target register read/write
```

## 주의점

| 오해 | 정리 |
| --- | --- |
| immediate bit가 섞인 것은 임의 배치다 | sign bit, register field, source bit 공유를 기준으로 hardware 배선을 줄인 결과다 |
| address range가 곧 실제 RAM 크기다 | address space의 배치 약속일 뿐이고 실제 장치가 연결된 범위만 동작한다 |
| bus에 신호가 뿌려지면 모든 장치가 반응한다 | select가 활성화된 장치만 의미 있게 반응한다 |
| APB도 pipeline bus처럼 겹쳐 전송한다 | APB는 `PREADY`를 기다리며 한 transfer가 끝난 뒤 다음 transfer로 넘어간다 |

## 핵심 정리

RV32I CPU는 instruction을 decode해 register file, immediate generator, ALU, memory path를 움직인다.
memory access 단계에서 만들어진 address/data/read/write는 bus로 나가고, memory map과 decoder가 선택한 memory 또는 peripheral만 반응한다.
이 관점이 memory-mapped I/O와 SoC bus 구조를 이해하는 출발점이다.
